# Story: Purge committed default secrets + rotate the demo JWT secret

**ID:** 2026-06-13-audit-02-purge-committed-secrets
**Status:** Reviewed — GLM runtime-breakage pass complete (2026-06-17)
**Audit findings covered:** S3, S9
**Phase / Priority:** Phase 0 (Quick win — ≤1 day, low risk, high value; roadmap item #6)
**Depends on:** none

## Goal
Stop shipping working default secrets in version-controlled files, and stop sprinkling the dev DB credential across scripts/tests. After this story, `docker compose up` with no `.env` **fails fast with a clear "set X" message** instead of silently booting with a known `NEXTAUTH_SECRET`, the public Supabase demo anon JWT, and DB password `symbio_dev_password`. Scripts/tests read `DATABASE_URL` from the environment with no inline credential fallback.

## Context
**Broken state (confirmed against source 2026-06-13):**

1. **S3 — `docker-compose.yml` (dev) ships working defaults:**
   - `:14` `DATABASE_URL: postgresql://symbio:${DB_PASSWORD:-symbio_dev_password}@db:5432/symbio?schema=public`
   - `:15` `NEXTAUTH_SECRET: ${NEXTAUTH_SECRET:-dev_secret_change_me_in_production}`
   - `:10` and `:25` `NEXT_PUBLIC_SUPABASE_ANON_KEY: ${...:-eyJhbGci…supabase-demo…I0}` — the well-known Supabase **demo** anon JWT (`iss: supabase-demo`, signed by the public `super-secret-jwt-token-with-at-least-32-characters`).
   - `:43` Postgres `POSTGRES_PASSWORD: ${DB_PASSWORD:-symbio_dev_password}`.
   `docker compose up` with no `.env` runs all of these. A demo anon key implies the paired stack may still validate the demo JWT secret (→ forge `service_role`).
   - **CORRECTION (live Codex 2026-06-13): the "known `NEXTAUTH_SECRET` lets anyone forge session tokens" claim is FALSE for the live code.** Cookie auth is **Supabase**, not NextAuth — `getTenantContext` calls `supabase.auth.getUser()` (`tenantContext.ts:89`), and there are **zero** `next-auth`/`getToken`/`NEXTAUTH_SECRET` references in `src/` (confirmed by grep). `NEXTAUTH_SECRET` is a **stale/dead** security target in the app runtime (it is still validated by `docker-entrypoint.prod.sh:24` and checked in `scripts/deploy-prod.sh:40`, so it is config-live but not auth-bearing). So requiring it in compose is hygiene/startup-friction, **not** a session-forgery fix. The real auth secrets are the Supabase **JWT secret** + **service-role key** (rotation step), not `NEXTAUTH_SECRET`. Keep the compose `${VAR:?}` change for hygiene, but do not justify it as "stops session forgery".
   - **Note (scoping):** `docker-compose.prod.yml` already does this correctly — `:23` `${DB_PASSWORD}` / `:24` `${NEXTAUTH_SECRET}` have **no** `:-default` (confirmed), with header comments instructing operators to `openssl rand -base64 32`. So this finding is specifically about the **dev** compose file. Both compose files are tracked in git (confirmed `git ls-files`).

2. **S9 — hardcoded DB credential + demo JWT in scripts/tests (all confirmed):**
   - `scripts/seed-demo-experiments.ts:26-28`: `const DATABASE_URL = process.env.DATABASE_URL || "postgresql://symbio:symbio_dev_password@localhost:5432/symbio?schema=public";`
   - `scripts/seed-agent-personas.ts:22-24`: identical inline fallback.
   - `scripts/fix-env.js:~40`: generates an `.env` with `DATABASE_URL="postgresql://symbio:symbio_dev_password@…"` and `DB_PASSWORD="symbio_dev_password"` baked in.
   - `tests/e2e/global-setup.ts:20-23`: probes ports `[54351, 54321]` with anon keys array including the same Supabase **demo** JWT `eyJhbGci…supabase-demo…I0`.
   - **(live Codex 2026-06-13) `scripts/test-agent-contract.sh:12`** defaults `TOKEN` to a real-looking committed **`skb_live_11158466…`** agent API key. This is squarely an "iOS/direct agent contract" credential and directly contradicts the "purge committed default secrets" goal — yet it was missing from the affected-file list. **Must be purged** (require the token via arg/env, no committed `skb_live_` default).
   - **(live Codex 2026-06-13) `.env.example`** itself ships the weak credentials: `:13` `DATABASE_URL=…symbio:symbio_dev_password@…` and `:17` `DB_PASSWORD="symbio_dev_password"`. The original story said example files are "safe to edit" but omitted `.env.example` from the affected-file list as a confirmed sprawl site. The example is what teaches new contributors the weak password — it must carry placeholders, not the literal `symbio_dev_password`.

**Why it matters:** Credential sprawl makes the weak password the path of least resistance everywhere; "rotate the DB password" silently breaks tooling, so the weak default persists. The committed demo anon JWT is the strongest signal that the live Supabase stack may still use the demo JWT secret (the audit's top UNVERIFIED item).

**Affected files:** `docker-compose.yml`, `scripts/seed-demo-experiments.ts`, `scripts/seed-agent-personas.ts`, `scripts/fix-env.js`, `tests/e2e/global-setup.ts`, **`scripts/test-agent-contract.sh` (committed `skb_live_` token — live Codex)**, **`.env.example` (ships `symbio_dev_password` + missing sync keys — live Codex)**. (`.env.local` is gitignored and hook-blocked from reading; out of reach and not the issue.)

JWT-secret rotation therefore does **not** make SKB validate iOS/Gateway JWTs — that is Story 01's job. This story's rotation language is scoped to "stop the demo key validating", not "secure the agent path".
