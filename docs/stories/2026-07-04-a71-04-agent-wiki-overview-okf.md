## Problem / motivation
SKB already has the raw materials for an agent-navigable overview — page hierarchy (`Page` self-reference tree, `spaceType`), a Chemistry KB hierarchy builder (`setupHierarchy.ts` creating Experiments/Archive/Reaction Types/Chemicals/Researchers/Substrate Classes/"Chemistry KB Index"), and an index-content generator (`src/lib/chemistryKb/indexPage.ts::generateIndexPageContent()`, plus `src/lib/sync/IndexGenerator.ts` and `src/lib/chemistryKb/aggregationRefresh.ts`) — but nothing enforces a **progressive-disclosure contract** an agent can rely on. `kb-query` (`src/lib/agent/kbQuery.ts`) already has a `depth: default|medium|deep` and `strategy: auto|rag|agentic` axis, and its `agentic` strategy path (search around `kbQuery.ts` L587-L800, expanding via `extractRelevantContent` at increasing depth and following linked/backlink pages) clearly assumes there's a cheap, small "orientation" layer to start from and richer content to drill into — but today's `generateIndexPageContent()` output isn't guaranteed to be that: it can grow arbitrarily large as experiments accumulate, with no explicit "one line per experiment, drill deeper on demand" discipline, and there is no documented contract telling an agent (or a future kb-query strategy) how to navigate cheaply before paying for a deep read.

The owner has pointed at `llm-wiki-builder/` (a separate OKF prototype) as the pattern to imitate: a root `index.md` with **one bullet per concept** (`- [Title](./concepts/slug.md) — <description> _(tags: …)_`), one file per concept under `concepts/`, cross-linked via relative paths, regenerated on every change (`wiki_manager.py::regenerate_index()`, `wiki_manager.py:97`), with an explicit two-tier agent API: `index_summary()` (cheap one-liners) then `full_contents()` (deeper bodies, ~40k char cap) — see `enrichment_agent.py`/`okf_models.py` for how an LLM consumes that shape.

## Proposed change
Bring the OKF two-tier contract into SKB's existing hierarchy machinery rather than building a parallel system.

**1. Root index page — "Experiments Index" (rework, not a light extend).** `setupHierarchy.ts` already creates a "Chemistry KB Index" page (~L402). **Important reality check (verified):** `generateIndexPageContent()` in `indexPage.ts` is today a **synchronous, zero-argument function returning a hardcoded static string** (a generic agent-navigation guide + tag taxonomy — it does not query the DB and lists no experiments). Making it list experiments is therefore a **signature-and-behavior change**, not a light extension: it must become `async`, take a `tenantId` (and a Prisma handle), query the experiment pages under the Experiments/Archive folders, and **all its callers must be updated** to `await` it and pass tenant context. Preserve the existing static guide as the "How to read this KB" preamble (see §3) rather than deleting it. Emit, for every active (non-archived) experiment page, exactly one line in the OKF bullet format:
```
- [Reduction of 4-nitrophenol (EXP-2025-0001)](./experiments/exp-2025-0001) — Pd/C catalyzed, yield 87%, status: complete _(tags: reduction, researcher:jsmith, 2026-06-30)_
```
The one-liner's description clause is sourced from the page's own metadata (title + a short derived summary — reuse the `exec_summary` field a71-02 introduces if present, falling back to the first ~120 chars of the page body otherwise) — never the full body. Archived experiments get their own line under a separate "Archive" heading (mirroring the existing Experiments/Archive split in `setupHierarchy.ts`), so agents can distinguish "currently relevant" from "historical" at the index layer without opening either page.

**2. Regeneration trigger — idempotent + concurrency-guarded.** Every sync event that changes an experiment page (lifecycle create/update/archive/restore from a71-01, content_update from a71-02/a71-03) enqueues an index regeneration. Reuse `IndexGenerator.ts` and `aggregationRefresh.ts`'s existing pattern — the reconciliation puller (`GET|POST /api/sync/reconcile` → `reconciliationSync.ts`) already has to solve "don't run two reconciliations at once" (`isReconciliationRunning()`, `getActiveSyncId()` in `reconciliationSync.ts` L77-L93); mirror that exact guard shape for index regeneration (`isIndexRegenerationRunning()` / a mutex keyed by tenant) so a burst of sync events doesn't fire N concurrent regenerations racing to write the same index page. Regeneration must be idempotent: running it twice in a row with no intervening changes produces byte-identical output (a straightforward property test). Every trigger call site (the sync-event handlers wired in `aggregationRefresh.ts`) must pass the `tenantId` explicitly through from the originating sync event to the regeneration call — never resolve it implicitly (e.g. from a shared/global context) — so a malformed or cross-tenant event can't cause one tenant's index to be regenerated with another tenant's data; verified `scheduleAggregationRefresh(tenantId, ...)` already takes `tenantId` as its first parameter (`aggregationRefresh.ts:66`), so the index-regeneration trigger should follow that same explicit-parameter convention rather than introducing an implicit lookup.

**3. Per-experiment page structure — standard shape for progressive disclosure.** Combine with a71-02's fixed section order: `# Title` → `## Executive Summary` (the cheap layer — this is what a `default`-depth `kb-query` should read) → `## Notebook Narrative` / `## ExpTube Analysis` / `## KB Notes` (the expensive layer — read only at `medium`/`deep`). Document this contract explicitly **inside the Chemistry KB Index page itself** (a short "How to read this KB" preamble, generated once and preserved across regenerations) so an agent landing on the index without any external documentation still learns the two-tier contract: *"Each entry below is a one-line summary. Follow its link and read only the Executive Summary section for a quick answer; read the full page for depth."* This mirrors `llm-wiki-builder`'s `index_summary()` → `full_contents()` split conceptually, expressed as a documented reading convention rather than a separate API (SKB already exposes the content via `GET /api/agent/pages/[id]` and `/api/agent/pages/tree` — no new transport is needed, just the discipline in what's returned).

**4. Exploit by kb-query's `agentic` strategy.** No code change required in `kbQuery.ts` for this story, but document explicitly (in this story and in a code comment added to `kbQuery.ts` near the agentic-strategy block, ~L669-L800) that the index page is now a reliable **cheap first hop**: the agentic strategy's existing pattern of "grab a starting page, look at depth, expand to linked/backlinks" gets measurably cheaper when the starting page for a broad query is the Experiments Index rather than a full experiment page, because the index's one-liners are bounded in size regardless of how much content a linked experiment page holds. This is worth calling out as a natural future optimization (route broad/discovery-style queries at `default` depth to the index page first) but implementing that routing change is optional/out of scope for this story — see below.

**5. Stale-index protection.** Same idempotency + concurrency guard as (2) covers "regenerate is safe to call redundantly." Additionally, timestamp the index page's own metadata with `lastRegeneratedAt` and the `correlation_id` of the sync event that triggered it, so a debugging session can answer "is this index current relative to the last known sync event" without re-running regeneration to find out.

## Out of scope
- Actually routing `kb-query`'s strategy selection through the index page for broad queries (documented as a future optimization, not implemented here).
- A dedicated `index_summary()`/`full_contents()` API distinct from the existing `/api/agent/pages/[id]` + `/api/agent/pages/tree` routes — the two-tier contract is expressed via page structure and documentation, not a new transport.
- Per-concept files outside the Chemistry KB domain (e.g. non-experiment pages) — this story scopes to the Chemistry KB Index / Experiments hierarchy only, matching what `setupHierarchy.ts` already covers.

## Acceptance criteria
1. `generateIndexPageContent()` emits exactly one bullet line per active experiment page, in the OKF format (`[Title](link) — description _(tags: ...)_`), and a separate section for archived experiments.
2. Running index regeneration twice with no intervening page changes produces byte-identical output. **Because the body is now DB-driven (not static), byte-stability requires a deterministic sort** — order experiments by a stable key (e.g. `position`, or `externalId`/`createdAt`) rather than relying on default DB row order, which is not guaranteed stable across queries.
3. Two sync events arriving within the same short window trigger exactly one regeneration pass, not two concurrent ones (verified via the concurrency-guard mechanism, mirroring `isReconciliationRunning()`'s test pattern).
4. The index page includes a preserved "How to read this KB" preamble describing the two-tier (summary-then-detail) reading contract, surviving regeneration (i.e. it's part of the generator's fixed output, not manually re-added each time).
5. The index page's metadata records `lastRegeneratedAt` and the triggering `correlation_id` after each regeneration.
6. An experiment's one-liner description reflects its `exec_summary` when present (post a71-02) and falls back gracefully to a body-derived summary when absent, without erroring on pages still in the empty-scaffold state.
7. Existing Chemistry KB Index / hierarchy tests (if present) continue to pass — the index content format changes, but the hierarchy structure (`Experiments`/`Archive`/`Reaction Types`/etc. as sibling pages) is untouched.

## Regression risks
- **Existing "Chemistry KB Index" consumers** (any UI or agent flow that reads the current index format) could break if the bullet format changes shape unexpectedly. Mitigate: keep the page's `externalId`/id stable, only change body content; audit for any code that parses the index body by string-matching the old format.
- **Regeneration triggered too eagerly** (e.g. one per lifecycle event rather than coalesced) could create write contention on a single page under sync bursts (this is exactly what a71-07's 8-10-experiment seed run will stress-test). Mitigate: the concurrency guard (AC3) plus considering a short debounce window, matching a71-02's 30s coalescing pattern, if bursts prove to be a real problem in testing.
- **Very large KBs make even the one-liner index big.** Not a concern at current/expected scale (dozens to low hundreds of experiments) but flag it: if the index itself grows past a comfortable single-page-read size, the two-tier contract would need a second level (index-of-indices) — explicitly out of scope now, noted for a future story if usage grows.
- **The mirrored concurrency guard is in-process only** (a single module-level `activeSyncId` in `reconciliationSync.ts`, not per-tenant, not cross-process). Under multiple Next.js workers, two workers can each pass the guard and race to write the same index page. Mitigate at current single-process scale by reusing the existing pattern; for multi-worker, a shared lock (DB advisory lock / Redis) is required — flagged as an owner decision, not silently assumed solved.
- **Non-deterministic row order breaks byte-stable idempotency** once the generator is DB-driven. Mitigate: always `ORDER BY` a stable key; never rely on implicit row order.
- **Regeneration triggers must carry `tenantId` explicitly, never implicitly.** A trigger call site that resolves tenant context from ambient state rather than the originating sync event's own `tenantId` risks regenerating (or leaking data into) the wrong tenant's index page. Mitigate: thread `tenantId` as an explicit parameter through every call site, mirroring `scheduleAggregationRefresh(tenantId, ...)`'s existing convention (`aggregationRefresh.ts:66`).
- **Brief read/write inconsistency during regeneration.** An agent calling `/api/agent/pages/tree` or reading the index page mid-regeneration could see a partially-written or momentarily-stale index. Not a correctness bug (the concurrency guard prevents concurrent *writers*, and a page read is a separate, unlocked read path), but worth documenting as an accepted eventual-consistency window rather than assuming reads are always perfectly fenced against an in-flight regeneration.
