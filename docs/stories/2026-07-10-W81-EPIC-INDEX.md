## Owner intent this batch encodes (from the 2026-07-10 brain-dump)
Combine two ideas into one system with **two layers**:
1. A fast, low-latency **Wiki** of digested conclusions you normally read directly (cheap retrieval).
2. An **immutable raw-source** layer underneath it. Every wiki claim links back to its exact original source, so you can always "pull it from the raw data again" and re-examine. Raw input is **never modified**; only the *conclusions drawn from it* change as the data underneath moves.
3. A **"sleeping" background process**: a cheap model runs 24/7 locally, going through new/existing sources, flagging which wiki entries are stale and tagging which new sources feed which articles. Only when it finds something genuinely worth updating does it **escalate to the expensive frontier model** for the actual re-synthesis.
4. **Version control** (GitHub-style) of wiki versions — you can go back and ask "what did the wiki say at time T".
5. Product split: the **Wiki** is personal knowledge / best-practice; the **Brain** is institutional knowledge teams retrieve from. Best/most-capable model *authors*; a cheap model *retrieves*.
6. Visualize it all in the knowledge-graph view.

## Deferred owner decision (routed to story review)
**Provenance granularity** — claim/span-level vs document-level. Both were designed; the recommendation from research + `/story` review is folded into **W81-A** (see its "Granularity decision" section). Headline: **chunk/paragraph-level with quote-text + SHA hash as the durable anchor, char offsets as a re-locatable hint, sentence-level reserved for high-stakes pages.** This is the NotebookLM/ReClaim operating point — best value-for-effort, and the quote-hash makes citations tamper-evident and re-anchorable when a source is re-parsed.

## Key architecture decisions (grounded in research — see per-story detail)
1. **Provenance = chunk-level + quote-hash anchor** (NotebookLM/ReClaim/eTracer). Char offsets are a hint, not the source of truth; re-find by hash on re-ingest. Store `supports`/`contradicts` from day one — contradiction edges are the trigger for the updater.
2. **Bitemporal supersession, not deletion** (Zep/Graphiti, arXiv:2501.13956). Four timestamps per claim edge. On new source: embed → pull semantically-near claims → LLM contradiction check → set `t_invalid`/`tx_expired` on the loser, insert the winner. **Graphiti's edge model + invalidation loop is the single most reusable prior art — it runs on Postgres, no Neo4j needed.**
3. **Time-travel = Postgres history tables**, not git. SKB already has `DocumentVersion` (per-page) + the A70 version-history UI; extend that rather than adopting git-Markdown (poor query substrate) or Dolt (operational split). Optionally mirror rendered pages to git later for a human-review PR surface.
4. **Updater = local-7B structural triage → frontier synthesis → human approval** (Slite pattern + arXiv:2606.26511). Staleness detection is **structural (`(subject,relation)` collision), not embedding-similarity** — cheap, deterministic, runnable 24/7 by a 7B model. Frontier only for contradiction *resolution* + re-synthesis. RouteLLM economics: ~85% cheap routing → ~half the frontier-everywhere cost.
5. **Retrieval = hybrid pgvector + tsvector + rerank over wiki-then-chunks.** The wiki layer IS the persisted synthesis that GraphRAG pays $$ to recompute per query — so teams read a pre-computed, cited page at near-zero marginal cost. **Skip full Microsoft GraphRAG**; adopt LazyGraphRAG selectively only if specific multi-hop query classes measurably underperform.
6. **Adapt research to SKB's actual stack:** it's Next.js 16 / Prisma 7 / **Postgres 18** (not Supabase-for-data; Supabase is auth-only against ExpTube :54341). pgvector runs on Postgres 18. Bitemporal + history via Prisma-managed tables + triggers. Enrichment engine is TypeScript (a71-13 already ports it). No RLS → every query carries `tenantId`.

## Cross-story invariants surfaced by review (record before implementing)
1. **Every `Page.properties` writer must use `jsonb_set` on its own subkey — never a full-column Prisma JSON write** (GLM). A full-column write racing a `jsonb_set` write still loses data if it lands second, so the "single owner of `knowledgeStatus`" guarantee (C1) only holds if a71-13 and all other `properties` writers also use `jsonb_set`. This is an **a71-13 coordination point**, not a W81-local fix.
2. **`pruneOldVersions` (`versioning.ts`) must skip `DocumentVersion` rows referenced by any `Claim.documentVersionId`** (GLM) — otherwise `onDelete: Restrict` FK-violates and rolls back user saves once a cited page passes the 100-version retention limit. Lands with whichever story adds `Claim.documentVersionId` (A2/B1).
3. **Migration order is strict + each migration immutable once consumed:** A70-rebased → a71-13 (`Page.properties`, `IngestLedgerEntry`) → W81-A1 (`Source`/`SourceChunk`, pgvector) → W81-A2 (`Claim`/`ClaimEvidence`, `EnrichJob`) → W81-B1 (bitemporal + `Source.eventDate`) → W81-C1 (`TriageFinding`/`SourceRelevance`/`TriageRun`).
4. **No RLS anywhere** — every query, and every FK connect inside a transaction, carries an explicit `tenantId` consistency check; soft-deleted (`deletedAt`) rows excluded from all scans/retrievals.
5. **The async worker/job substrate is new to SKB** — there is no durable job infra today (only `aggregationRefresh.ts`'s in-process `Map`+`setTimeout`). A2 introduces a durable `EnrichJob` table; C1 runs as a bounded, cron-scheduled `agent-sweep`-style sweep, not a daemon.
