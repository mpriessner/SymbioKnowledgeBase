# Story: Close the og-metadata SSRF and the middleware "any-Authorization-header passes" hole

**ID:** 2026-06-13-audit-03-ssrf-and-middleware-bearer-passthrough
**Status:** Reviewed — awaiting approval
**Audit findings covered:** S5
**Phase / Priority:** Phase 1 (Close the auth cliffs; roadmap item #16)
**Depends on:** none (independent of Story 01, but both touch `src/middleware.ts` — coordinate the merge; see Risks)

## Goal
Make the bookmark-preview endpoint safe and stop the middleware from treating the mere *presence* of an `Authorization` header as proof of auth. After this story, `POST /api/og-metadata` (a) requires real authentication and (b) refuses to fetch private/loopback/link-local/cloud-metadata addresses or non-`http(s)` schemes, so it can no longer be used as an unauthenticated server-side request proxy into the internal network.

## Context
**Broken state (confirmed against source 2026-06-13):**

1. **Unauthenticated SSRF (`src/app/api/og-metadata/route.ts`):** The route validates only `z.string().url()` (`:5-7`), then `fetch(url, …)` at `:33` with no allow/deny list — no block of `localhost`, RFC1918 (`10/8`, `172.16/12`, `192.168/16`), link-local `169.254.0.0/16` (incl. `169.254.169.254` cloud metadata), or non-http schemes beyond what `.url()` permits (`.url()` accepts e.g. `http://`, `https://`, and notably does not block `http://169.254.169.254/...` or `http://host.docker.internal:54341/...`). The route has **no auth wrapper** (no `withTenant`/`withAgentAuth` — confirmed: the file exports a bare `POST`).
2. **Middleware bearer-passthrough (`src/middleware.ts:27-33`):**
   ```ts
   if (pathname.startsWith("/api/") && request.headers.get("authorization")) {
     return NextResponse.next();   // any header *value* passes
   }
   ```
   Any request to any `/api/*` path carrying *any* `Authorization` header value (e.g. `Authorization: x`) skips the Supabase session check entirely. The intent was "let API-key auth through so the route can validate it", but it lets a garbage bearer reach routes that don't themselves authenticate — like `og-metadata`. (`/api/og-metadata` is not a `PUBLIC_PATH` and has no dot in its pathname, so it is otherwise gated by middleware — the passthrough is the only way an unauthenticated caller reaches it.)

**Why it matters / failure scenario:** `POST /api/og-metadata` with header `Authorization: anything` and body `{"url":"http://169.254.169.254/latest/meta-data/iam/security-credentials/"}` makes the **server** fetch cloud metadata and returns the response text inside the JSON `title`/`description` (`:50-55`, `:59-64`) — exfiltrating internal data. Same trick reaches `http://host.docker.internal:54341/...` (the internal Supabase/Kong) and any other localhost service. Server binds `0.0.0.0` and is exposed over Tailscale, so the caller need not be local.

**Affected files:** `src/app/api/og-metadata/route.ts`, `src/middleware.ts`. New: a small SSRF-guard util (e.g. `src/lib/security/ssrfGuard.ts`) + tests.

## Acceptance Criteria
- [ ] `POST /api/og-metadata` with no valid session/API key returns **401/redirect** (it is no longer reachable via a junk `Authorization` header). The S5 unauth reproduction fails.
- [ ] `POST /api/og-metadata` with a body URL whose resolved host is loopback (`127.0.0.0/8`, `::1`, `localhost`), private (`10/8`, `172.16/12`, `192.168/16`, `fc00::/7`), or link-local (`169.254.0.0/16`, `fe80::/10`) returns a **422/400** error and performs **no** outbound fetch. Covers `169.254.169.254` and `host.docker.internal` (which resolves to a private/host-gateway address).
- [ ] Non-`http(s)` schemes (`file:`, `ftp:`, `gopher:`, `data:`) are rejected before any fetch.
- [ ] A legitimate public URL (e.g. `https://example.com`) still returns OG metadata (happy path preserved).
- [ ] The "garbage `Authorization` header reaches an unauthenticated route" hole is closed **primarily by wrapping every reachable `/api/*` route in real auth** (og-metadata being the cited gap). **The middleware bearer-passthrough is NOT deleted** (see revised step 4 + Gemini feedback): deleting it would 307-redirect legitimate Bearer-only clients (`/api/agent/*`, `/api/sync/*`) to `/login`, because `updateSession` only validates cookies. Instead the passthrough is kept (so route handlers can validate the bearer themselves) and the security property is "every route behind the passthrough validates its own credential" — verified by an inventory of all `/api/*` routes confirming none is unwrapped.
- [ ] Redirect-following does not defeat the guard: if the initial host is public but the server follows a redirect to a private/link-local host, the fetch is aborted (see Implementation step 3). 
- [ ] **(live Codex) The no-private-fetch guarantee is scoped honestly to the DNS-resolution-time check, NOT an absolute invariant.** Because `assertUrlIsFetchable` resolves via `dns.promises.lookup` and then `fetch(safeUrl)` lets undici resolve the hostname *again*, a DNS-rebinding host (public at check-time, private at connect-time) has a residual TOCTOU window. So the AC reads "rejects URLs whose host **resolves at validation time** to loopback/private/link-local, and refuses redirects" — it must **not** claim "the server never performs an outbound fetch to a private address" unless IP-pinning is implemented (resolve → connect by the validated IP with a `Host` header, or a custom `undici` lookup/agent). If IP-pinning is deferred (the documented default), the acceptance language and any security note must state the residual rebinding risk explicitly rather than over-promise.
- [ ] **(live Codex) This story does NOT close the "any Bearer reaches `withAgentAuth` as mock principal" cliff** (`agent/auth.ts:57-67`). Keeping the middleware passthrough is correct for reachability, but the story's phrase "every route behind the passthrough validates its own credential" is **only true once Story 01 lands** — today a non-`skb_` Bearer to `/api/agent/*` is accepted as `mock-user-id`/`DEFAULT_TENANT_ID`. The AC is reframed: this story closes the **og-metadata** unauth+SSRF hole and inventories the routes; the agent-path credential validation is Story 01's job and is explicitly out of scope here (cross-referenced, not silently assumed fixed).
- [ ] New tests cover: each blocked CIDR/scheme, the public happy path, the unauth rejection, and (if implemented) the redirect-to-private case. `npm test` green.

## Implementation Plan
1. **SSRF guard util (`src/lib/security/ssrfGuard.ts`).** Export `assertUrlIsFetchable(rawUrl: string): Promise<URL>` that:
   - Parses the URL; rejects non-`http:`/`https:` protocols.
   - Resolves the hostname to IPs via `dns.promises.lookup(host, { all: true })` and rejects if any resolved address is loopback/private/link-local/unique-local. Use a vetted check (e.g. the `ip` / `ipaddr.js` semantics, or hand-rolled CIDR tests against `net.isIP` results). Also reject literal `localhost` and `*.localhost`, and bare IPs in those ranges before DNS.
   - Returns the parsed `URL` on success; throws a typed error otherwise.
   - Verification: unit tests enumerating blocked/allowed inputs incl. IPv4-mapped IPv6 (`::ffff:169.254.169.254`).
2. **Apply guard + auth in `og-metadata/route.ts`.**
   - Wrap the handler so it requires authentication. Use the cookie/API-key path — `getTenantContext(req)` from `src/lib/tenantContext.ts` (throws `AuthenticationError` → map to `createAuthErrorResponse`), matching how other authed API routes resolve identity. (Bookmark preview is a UI action by a logged-in user, so `getTenantContext` is the right gate **for the only current caller** — `BookmarkView.tsx:33`, same-origin cookie fetch with **no** `Authorization` header.)
   - **(live Codex 2026-06-13) Two caveats this introduces — call them out, do not paper over:**
     - **Bearer takes precedence over cookies in `getTenantContext` and rejects an invalid Bearer *before* trying cookies** (`tenantContext.ts:41-52`): if `Authorization: Bearer …` is present it calls `resolveApiKey` and **throws 401** if the key is invalid, never falling back to the session cookie. So any future/browser wrapper that attaches a stale/malformed `Authorization` header to the og-metadata call breaks even with a valid session. The current caller sends no such header, so it's fine **today** — but a test must assert "malformed Bearer + valid cookie → 401" so this behavior is understood, not discovered later.
     - **The "API-key path" is only half-real — bcrypt-created keys do NOT work on `getTenantContext`.** `resolveApiKey` (`apiAuth.ts:53`) looks up by a **deterministic SHA-256** `hashApiKey(rawKey)`, but `/api/settings/api-keys/route.ts:36` stores keys as **`bcrypt.hash(key)`** (salted, non-deterministic) — so a settings-UI key can never match the SHA lookup. Settings-created bcrypt keys work in `withAgentAuth` but **not** in `getTenantContext`; SHA keys (`apiAuth.ts`) work in `getTenantContext` but are not what the settings UI mints. **Do not write an og-metadata "API-key auth" test without picking the correct key path** (SHA via `hashApiKey`), or it will fail for the wrong reason. This is a pre-existing inconsistency this story must not trip over (and is flagged to Story 06, which also touches API-key/audit paths).
   - Before `fetch` at `:33`, replace `const { url } = parsed.data;` usage with `const safeUrl = await assertUrlIsFetchable(parsed.data.url);` and fetch `safeUrl`. On guard rejection return `errorResponse("VALIDATION_ERROR", "URL not allowed", …, 422)`.
   - Verification: integration test hitting the route with blocked/allowed URLs and with/without auth.
3. **Defeat redirect-based bypass.** Set `fetch(safeUrl, { redirect: "manual", … })` and, if a 3xx with a `Location` is returned, re-run `assertUrlIsFetchable` on the resolved `Location` and follow at most N (e.g. 3) hops, re-validating each. Alternatively (simpler, acceptable): `redirect: "error"` to refuse all redirects for bookmark previews. Pick the simpler `redirect: "error"` unless preview quality regresses. Verification: test a public URL that 302s to `http://169.254.169.254` → aborted.
4. **Do NOT delete the middleware passthrough — wrap the routes instead (REVISED per Gemini integration review).** The original "delete the block" idea is **wrong**: `src/middleware.ts:27-33`'s "`/api/*` + any `authorization` ⇒ next()" shortcut is what lets Bearer-only clients reach their route handlers. `updateSession` (`src/lib/supabase/middleware.ts:43-73`) validates **only cookies**; a Bearer-only request (no cookie) returns `user: null`, so deleting the shortcut makes the middleware 307-redirect `/api/agent/*` and `/api/sync/*` to `/login` (`middleware.ts:38-42`) — breaking the live Gateway + sync paths. The route handlers (`withAgentAuth`, `withTenant`→`getTenantContext`, sync `authenticateSync`) all validate the bearer **themselves**, but only **after** middleware yields.
   - **Correct fix:** keep the passthrough; close the actual hole (a garbage bearer reaching an *unauthenticated* route) by ensuring **every** route reachable via the passthrough validates its own credential. og-metadata is the cited unwrapped route — step 2 wraps it. 
   - **Inventory step (required):** enumerate all `src/app/api/**/route.ts` and confirm each either is a `PUBLIC_PATH`, or is wrapped in `withTenant`/`withAgentAuth`/`withAdmin`/`authenticateSync`, or otherwise authenticates. Any other "naked" route (bare exported `POST`/`GET` with no auth) is a latent twin of the og-metadata bug. **(live Codex MUST: the inventory must *classify* every naked route as intentional-public vs accidental, not just assert "none exist".)**
   - **Inventory result (completed during this review — 2026-06-13):** the only bare-export routes with no auth wrapper are: **`og-metadata` POST** (the cited bug — wrapped by step 2); **`openapi.yaml` GET** + **`agent/openapi.yaml` GET** (intentional public API docs — note: these reach the client via the middleware's `pathname.includes(".")` shortcut at `middleware.ts:22`, **not** via `PUBLIC_PATHS` and **not** via the bearer-passthrough); **`health` GET** (intentional, also a `PUBLIC_PATH`); **`auth/register` POST** (public by the `/api/auth` `PUBLIC_PATH` prefix — but note it creates a `User`+`Tenant` for any caller supplying a `supabaseUserId` with **no verification the caller owns that Supabase id**; this is a provisioning-trust concern that belongs with Story 04, flagged there); and the **`OPTIONS` preflight handlers** on `sync/experiments:57` and `sync/reconcile:32` (unauthenticated by design — CORS preflights carry no credentials; Story 05 tightens their `*` origin). All non-`agent`/`sync` data routes are `withTenant`/`withAdmin`-wrapped (e.g. `users/route.ts` is `withAdmin`). **Conclusion: og-metadata is the only *accidental* naked route; no additional route needs wrapping in this story.** (Confirmed callers today: og-metadata is called same-origin-with-cookies from `src/components/editor/nodeViews/BookmarkView.tsx:33`, so wrapping it in `getTenantContext` does not break the editor.)
   - **Optional hardening (not a replacement for wrapping):** the passthrough could additionally require the header to match `^Bearer\s+\S+` (reject `Authorization: ` empty/garbage-format) to shave the most trivial probes, but this is cosmetic — the real guarantee is that every route validates its own credential. Do not rely on header-shape checks for security.
   - Verification: test that a valid `skb_` key to `/api/agent/*` still works end-to-end (no 307); **(live Codex) ALSO test a valid `SYNC_SERVICE_KEY` Bearer to `/api/sync/experiments` AND `/api/sync/reconcile` survives the passthrough (these use `authenticateSync()`, NOT `withTenant`/API keys — `sync/experiments:30`, `sync/reconcile:11` — so narrowing the passthrough to only `skb_` keys would break them; the test must cover the sync contract, not just `/api/agent/*`);** test that `/api/og-metadata` with `Authorization: junk` and no cookie now returns 401 from the *route's* auth wrapper (not from middleware); test that the editor's cookie-authed call still succeeds; **(live Codex) test "malformed `Authorization` Bearer + valid session cookie → 401" so the Bearer-precedence behavior in `getTenantContext` is locked in.**

## Risks & Open Questions
- **Merge coordination with Story 01:** both edit `src/middleware.ts`. Story 01 changes the *dev/ADMIN fallback*; this story removes the *bearer-passthrough*. They are in different blocks but adjacent — land one, rebase the other. No logical conflict.
- DNS rebinding: a host that resolves to public at check time but private at fetch time. `dns.lookup` + immediate `fetch` leaves a tiny window; pinning the resolved IP into the fetch (connecting by IP with a `Host` header) closes it but is more complex. Mitigation: accept the small residual risk for a bookmark-preview feature, or pin the IP. `NEEDS USER INPUT:` is IP-pinning warranted, or is DNS-time validation + `redirect:"error"` sufficient? Assumption: DNS-time validation + no-redirects is sufficient for this low-value feature.
- Requiring auth on `og-metadata` changes its contract: any client calling it unauthenticated (does the editor call it with cookies? It should, since it's a logged-in editor action) will now need a session. Verify the TipTap bookmark/link-preview caller sends cookies. Assumption: it does (same-origin fetch from the authenticated app).
- `host.docker.internal` resolves to the host-gateway address (often a private/`172.x`); ensure the guard's DNS resolution catches it in the container runtime.

## Out of Scope
- The mock-auth / ADMIN-everyone fallbacks (Story 01).
- CORS on the sync endpoints (Story 05).
- Other routes' SSRF surfaces (none cited; og-metadata is the only user-URL-fetching route in the audit).

### Reviewer Feedback / Gemini (integration) — live (gemini-3.1-pro-preview, plan mode)
Integration breakage:
- **API client breakage (Critical):** removing the middleware "`Authorization` present ⇒ next()" shortcut (`src/middleware.ts:27-33`) **breaks all non-browser clients** (agents, sync CLI, cross-app). `updateSession` only validates cookies (`src/lib/supabase/middleware.ts:43-73`); a Bearer-only request returns `user: null` → middleware 307-redirects to `/login` (`middleware.ts:38-42`). The shortcut must stay so route handlers (`withAgentAuth`/`withTenant`) can validate the bearer themselves; close the "garbage bearer" hole by wrapping **all** protected routes (incl. og-metadata) instead.
- **Frontend TipTap/Bookmark (no breakage):** og-metadata is called via same-origin `fetch` in `BookmarkView.tsx:33`; the browser includes session cookies, so `getTenantContext` resolves identity from cookies. No break.
- **Supabase config (no breakage):** `getTenantContext` has the `isSupabaseConfigured()` dev fallback (`tenantContext.ts:118-126`), so adding it to og-metadata doesn't force Supabase config in local dev. (NB: Story 01 narrows that fallback to require `ALLOW_DEV_AUTH=1`.)
- **Agent/Sync routes validate their own tokens** (`agent/auth.ts:28`, `withTenant.ts:31`) **but depend on the middleware letting them through** — removing the shortcut makes them unreachable for Bearer-only clients.

**Disposition:** This overturned the original plan's "delete the passthrough" approach — step 4 rewritten to KEEP the passthrough and close the hole by wrapping every route + an inventory step. The biggest single correction in this story.

### Reviewer Feedback / Kimi (runtime) — FALLBACK: self (Opus 4.8); reason: Kimi CLI returned HTTP 429 usage-limit.
Runtime breakage / missed bugs:
1. **DNS-rebinding TOCTOU is real but low-stakes here.** `dns.lookup` then `fetch` leaves a window where a host resolves public at check-time and private at connect-time. For a bookmark-preview feature the residual risk is small; the bullet-proof fix (resolve → pin the IP → connect by IP with a `Host` header) is more code. The story already flags this as a NEEDS USER INPUT with a sane default (DNS-time validation + `redirect:"error"`). Combined with `redirect:"error"`, the most common SSRF chains (public→302→169.254.169.254) are closed. Acceptable.
2. **`redirect:"error"` is the simplest correct choice** and avoids the per-hop re-validation complexity; the story already prefers it. Confirmed.
3. **`host.docker.internal` resolution inside the container:** the guard's `dns.lookup` will resolve it to the host-gateway address (often `192.168/172.x` private), so the private-range check catches it — but only if the guard runs in the **same** runtime as the fetch (Node.js). The runtime-pin (Codex item 1) guarantees this. Cross-linked.
4. **No process-crash path** in the guard; on rejection it throws a typed error caught by the route's try/catch → 422. The existing `AbortController` timeout (`og-metadata route.ts:30-31`) is preserved. Confirmed safe.
5. **Edge case:** a URL like `http://0.0.0.0` or `http://[::]` (unspecified address) and decimal/octal/hex IP encodings (`http://2130706433/` = `127.0.0.1`) must be normalized before the CIDR check — `new URL()` + `net.isIP` on the resolved lookup handles most, but add explicit tests for these obfuscations. Folded into the test matrix note.
No runtime bug introduced; the dominant residual is the documented TOCTOU, mitigated by no-redirects.
