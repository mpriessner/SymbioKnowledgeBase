## Problem / motivation
Today three systems know about lab experiments and none of them agree on identity. ExpTube is the sync hub: `lib/actions/experiments.ts` fires `syncCreateToSKB`/`syncUpdateToSKB`/`syncArchiveToSKB`/`syncRestoreToSKB`/`syncDeleteToSKB` (ExpTube `lib/integrations/skb-sync-service.ts`) on create (~L271, L282), archive/restore/rename (~L369-L491), which POST to SKB's `POST /api/sync/experiments` (`src/app/api/sync/experiments/route.ts`). ExpTube also receives inbound lifecycle events at `app/api/sync/experiments/route.ts` (its own hub-inbound route) from ChemELN and relays them on to SKB — see the file's own header comment "EXPTUBE-200.4: Incoming Experiment Sync Endpoint... Hub relay: forwards to SKB". Both ends key everything off `experiments.eln_experiment_id` (e.g. `EXP-2025-0001`, TEXT unique per user) and an `origin` column (`exptube|chemeln|mobile`, added by migration `107_add_experiment_origin.sql`). SKB's receiver treats this ID as the idempotency anchor: `Page.externalId` is unique per tenant (`uq_pages_tenant_id_external_id` in `prisma/schema.prisma:224`), looked up via `findExperimentByElnId` (`src/lib/chemistryKb/experimentLookup.ts`).

Notebook (`chatbot-notebook-transparency-prototype-ui`, FastAPI on :8000) has no `eln_experiment_id`, no `origin`, and does not call any of this. Experiments are folders `experiments/<uuid4>/` created by `POST /api/experiments` (`backend/app.py:937-949`, calling `store.create_experiment()`), with matching lifecycle handlers `DELETE /api/experiments/{id}` (`app.py:1025-1026`), `POST /api/experiments/{id}/restore` (`app.py:1067-1068`), and `PATCH /api/experiments/{id}/rename` (`app.py:3236-3237`). None of these fire any outbound event. The one hook point that exists, `ToolContext.on_notebook_write` (`backend/tools.py:434`), is wired only to `_bump_reload_required` (`app.py:826`, injected at construction `app.py:254`) — a local polling-refresh bump, not a network call. Net effect: an experiment created in the notebook is invisible to ExpTube and SKB, and the notebook has zero record of experiments created upstream. Nothing in ExpTube's `origin` enum recognizes a notebook source either.

## Proposed change
Make the notebook a full peer in the sync mesh: allocate a shared ID per experiment, and fire lifecycle events at ExpTube's hub the same way ChemELN already does.

**1. ID allocation.** Add `eln_experiment_id: str | None` to the notebook's per-experiment metadata (stored in `notebook.ipynb` metadata dict alongside `experiment_id`, `name`, `description` — same location `store.create_experiment()` already writes to). Allocation policy: self-allocate with a `NOTEBOOK-YYYY-NNNN` prefix (avoids a synchronous dependency on ExpTube being reachable at creation time, which matters because the notebook runs disconnected on a laptop while ExpTube/SKB run on the Mac mini). Collision rule: NNNN is a per-day monotonic counter persisted in a small local JSON index file (`experiments/_id_allocator.json`, `{last_seq: int, date: "YYYY-MM-DD"}`), reset at UTC midnight; on counter-file corruption, fall back to scanning existing experiment folders for the max NOTEBOOK- suffix used that day plus 1. This mirrors the "self-allocate, don't block on network" spirit already present in the notebook's offline-first design (no DB, filesystem is truth). Do NOT attempt to ask ExpTube for an ID synchronously — that would turn experiment creation into a network-dependent operation, which nothing else in this codebase does.

**2. Lifecycle event dispatch.** Add a new outbound module `backend/sync_client.py` with a single function `send_lifecycle_event(action, eln_experiment_id, fields, correlation_id=None)` that POSTs to `{NOTEBOOK_SYNC_HUB_URL}/api/sync/experiments` with body matching ExpTube's existing `SyncRequest` interface (`app/api/sync/experiments/route.ts`): `{eln_experiment_id, action: "create"|"delete"|"restore"|"update", source: "notebook", correlation_id, fields: {name, title, status}}`. Auth: `Authorization: Bearer {NOTEBOOK_SYNC_SERVICE_KEY}` — ExpTube's hub-inbound route validates via `authenticateServiceRole()`, which currently expects `SUPABASE_SERVICE_ROLE_KEY`; confirm with the ExpTube owner whether notebook gets its own dedicated service credential or reuses this one (recommend a dedicated key scoped to `source=notebook` if ExpTube's auth check supports distinguishing callers — otherwise document the shared-secret risk in Regression risks below).

Wire the call sites:
- `create_experiment` (`app.py:937-949`): after `store.create_experiment()` succeeds and the ID is allocated, call `send_lifecycle_event("create", eln_id, {"name": name, "title": title})`.
- `delete_experiment` (`app.py:1025-1026`): call `send_lifecycle_event("delete", eln_id, {})`.
- `restore_experiment` (`app.py:1067-1068`): call `send_lifecycle_event("restore", eln_id, {})`.
- `rename_experiment` (`app.py:3236-3237`): call `send_lifecycle_event("update", eln_id, {"title": new_title})`.

All four calls are fire-and-forget (async task, do not block the HTTP response) — matches ExpTube's own `skb-sync-service.ts` pattern ("fire-and-forget, 10s timeout, DLQ fallback").

**3. Failure handling / offline tolerance.** The notebook runs on a laptop that is frequently offline from the Mac mini (where ExpTube/SKB live). A failed `send_lifecycle_event` call must never fail the local operation. On failure (timeout, connection refused, non-2xx), append the event to a local DLQ file `experiments/_sync_dlq.jsonl` (append-only, same hash-chained-ledger spirit as `audit.jsonl`), using an atomic tmp-file-plus-rename write so a mid-write crash never leaves a torn/partial JSONL line; validate the file's lines at startup and skip (log, don't crash on) any that fail to parse. Add a background retry task (poll every 60s while the FastAPI process is up) that drains `_sync_dlq.jsonl` in order, re-POSTing each entry and removing it on success; entries older than 7 days are logged as permanently failed and left in the file for manual inspection (no silent data loss). Use a short exponential backoff (e.g. starting at 5s, doubling to a 60s cap) between consecutive retries of the *same* entry rather than a flat 60s poll, so a hub that is flapping (briefly reachable, then not) doesn't get hammered with a retry storm. This is simpler than a real queue but matches the notebook's existing "no database, filesystem is truth" architecture.

**4. `origin` recognition.** ExpTube's `experiments.origin` is **not a Postgres enum** — it is a `TEXT NOT NULL DEFAULT 'exptube'` column guarded by a CHECK constraint `CHECK (origin IN ('exptube','chemeln','mobile'))` (verified: `supabase/migrations/107_add_experiment_origin.sql`; the migration's own comment reads *"Uses TEXT + CHECK instead of ENUM so future origins only need a constraint ALTER"*). Adding `notebook` therefore means a migration that **drops and re-adds the CHECK** with `('exptube','chemeln','mobile','notebook')` in the IN-list (plus a TypeScript union update), **not** a non-blocking `ALTER TYPE ... ADD VALUE`. **Failure mode if this is not done first (precise):** the hub-inbound create path builds `insertData.origin = body.source === 'chemeln' ? 'chemeln' : body.source` (verified `app/api/sync/experiments/route.ts` ~L124-131), so a notebook create arrives as `origin: 'notebook'`, the INSERT violates the CHECK, and the hub returns HTTP 500 `"Failed to create experiment"` (~L136-139) *before* it relays to SKB (`syncCreateToSKB` at ~L143). Consequence: with the CHECK un-patched, a notebook create produces **neither** an ExpTube row **nor** a SKB page — this is a hard precondition, not a soft "miscategorized" degrade. Recommended still: **yes**, notebook-originated experiments get an ExpTube row (not just relayed-through-to-SKB) — this lets the ExpTube UI link uploaded lab videos to the correct notebook experiment by `eln_experiment_id`. This is an ExpTube-side schema change; call it out explicitly to whoever picks up the ExpTube side (this story's scope is the notebook-side dispatch + the origin-CHECK request, not the ExpTube migration itself — see Affected repos & files). Also verify `experiments.user_id` is nullable for cross-platform creates — the hub insert sets no `user_id` (its own comment: *"user_id may be unknown for cross-platform creates"*); if a NOT NULL exists there, notebook creates fail for a second, separate reason.

## Acceptance criteria
1. Creating a new notebook experiment allocates a unique `eln_experiment_id` of the form `NOTEBOOK-YYYY-NNNN`, persisted in the experiment's metadata, surviving a process restart.
2. Two experiments created within the same UTC day never collide on `eln_experiment_id`, including when created within the same second (verified via the monotonic counter file, not wall-clock).
3. `create_experiment`, `delete_experiment`, `restore_experiment`, and `rename_experiment` each attempt a `send_lifecycle_event` call with the correct `action` and `eln_experiment_id`; a mocked ExpTube hub receives the expected payload shape matching `SyncRequest`.
4. When the ExpTube hub is unreachable (connection refused / timeout), the local operation still succeeds and returns normally to the caller — no user-visible error, no added latency beyond the configured request timeout for the fire-and-forget task.
5. A failed event is appended to `experiments/_sync_dlq.jsonl` and is retried on the next drain cycle; on success it is removed from the file.
6. Entries older than 7 days remain in the DLQ file and are logged at WARNING level on each drain cycle (not silently dropped, not endlessly retried at full frequency).
7. `NOTEBOOK_SYNC_HUB_URL` / `NOTEBOOK_SYNC_SERVICE_KEY` unset → sync dispatch is a no-op (logged once at startup), never a crash.

## Regression risks
- **Existing experiment folders lack `eln_experiment_id`.** A migration/backfill script must assign IDs to pre-existing experiments on first load, or all code paths reading this field must treat `None` as "not yet synced" rather than erroring. Mitigate: make the field optional everywhere it's read; add a lazy-backfill on next experiment access.
- **Shared secret risk if notebook reuses `SUPABASE_SERVICE_ROLE_KEY`.** That key currently authenticates ExpTube's own Supabase-internal server-to-server calls; broadening its use to an external process run on a different physical machine (a70-06's targeted hardening is specifically about tightening tenant-key binding — read that story before deciding this) increases blast radius if the notebook laptop is compromised. Mitigate: issue a dedicated, narrowly-scoped `NOTEBOOK_SYNC_SERVICE_KEY` on the ExpTube side instead of reusing the Supabase role key.
- **DLQ file growth if ExpTube is down for extended periods.** Mitigate: cap file size or rotate; the 7-day stale-entry log is a signal to intervene manually, not a silent no-op.
- **`origin` CHECK-constraint change on ExpTube is a schema migration on a live table.** Because it is a CHECK (not a Postgres enum), the migration must `DROP CONSTRAINT` then `ADD CONSTRAINT` with the widened IN-list, which re-validates every existing row against the new predicate under a lock. Mitigate: the new predicate is a strict superset (`'exptube','chemeln','mobile','notebook'`), so all existing rows already satisfy it; run it as a fast additive constraint swap, never removing/renaming existing values. This is strictly more involved than the "additive enum value" the earlier draft implied.
- **The notebook→SKB page depends on the ExpTube insert succeeding first** (hub relays to SKB only after its own row insert). See section 4: the SKB scaffold is *not* independent of the ExpTube-side CHECK fix. Downstream stories that assume a SKB page appears from a notebook create (a71-02 content, a71-07 seeding) inherit this precondition.
- **The per-day ID allocator assumes a single notebook process/laptop.** `experiments/_id_allocator.json` is local-filesystem state with no cross-process lock. If the notebook is ever run from two devices/processes against the same `EXPERIMENTS_ROOT` in the same day (unlikely in the current one-laptop-per-user usage pattern, but not structurally prevented), both could read the same `last_seq` and allocate the same `NOTEBOOK-YYYY-NNNN` ID. Mitigate: either document this as an accepted single-process constraint, or add a file lock (e.g. `fcntl`/`portalocker`) around the read-increment-write of the counter file.
