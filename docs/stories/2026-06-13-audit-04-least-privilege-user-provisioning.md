# Story: Stop auto-provisioning every new user as ADMIN of the shared tenant

**ID:** 2026-06-13-audit-04-least-privilege-user-provisioning
**Status:** Reviewed — awaiting approval
**Audit findings covered:** S4
**Phase / Priority:** Phase 1 (Close the auth cliffs; roadmap item #16 — least-privilege users)
**Depends on:** Story 01 (real auth) — recommended to land after, since the auth path and `ensureUserExists` are co-evolving; not a hard blocker

## Goal
A user who self-registers on the shared ExpTube Supabase must **not** automatically become an **ADMIN** inside SymbioKB's shared tenant with full **write/delete** over all seeded + synced experiment knowledge. After this story, a first-time SSO user is provisioned at **least privilege**.

> **IMPORTANT — design tension surfaced in review (Gemini integration):** This product is a **single shared knowledge base**. The seeded chemistry KB *and* all experiments synced from ChemELN/ExpTube live in `DEFAULT_TENANT_ID`, and the live voice agent's `kb-query` (`src/lib/agent/kbQuery.ts`) filters **strictly** by `tenantId` with no cross-tenant read-through (confirmed: `kbQuery.ts:159,165,193`). Therefore the audit's literal fix ("give every new user a fresh **personal** tenant") would make `kb-query` return **zero results** for every new SSO user and break the live demo, and would silo synced experiments where no new user can see them. **So the correct least-privilege fix here is NOT "personal tenant for everyone" — it is "stay in the shared tenant but drop new users from ADMIN/owner to MEMBER (read-only / non-destructive)".** Personal-tenant-by-default is offered only as an opt-in for a future multi-tenant posture. This reframing is the core review outcome for this story.

## Context
**Broken state (confirmed against source 2026-06-13):**

- `src/lib/auth/ensureUserExists.ts:24-43`: on first login the user's tenant is resolved as `user_metadata.tenantId ?? resolveDefaultTenant() ?? createPersonalTenant()`. Because `DEFAULT_TENANT_ID` is set in compose (`docker-compose.yml:19`, default `00000000-0000-4000-a000-000000000001`) **and** that tenant exists (seeded), `resolveDefaultTenant()` (`:89-95`) returns it for every new user — so `createPersonalTenant` is essentially never reached. The created `User` row is hardcoded `role: "ADMIN"` (`:36`), and a `TenantMember` is created with `role: "owner"` (`:54`) in that shared tenant.
- Auth is delegated to ExpTube's Supabase, which has **open signup** (the ecosystem audit: ExpTube `enable_signup` with confirmations disabled; this repo's own `supabase/config.toml:37,40` also shows `enable_signup = true`, though that stack is dead). So "anyone who can self-register on ExpTube" ⇒ "ADMIN of SymbioKB's shared tenant".

**Why it matters / failure scenario:** A stranger signs up on the shared ExpTube Supabase, navigates to (or has the Gateway/iOS reach) SymbioKB, and is silently provisioned as an ADMIN owner of the demo tenant — able to read and modify all seeded chemistry KB + every experiment synced from ChemELN/ExpTube. No role gating meaningfully restricts a non-owner because every new user *is* an owner/ADMIN.

**Interaction with Story 01:** Story 01 closes the *authentication* hole (who is let in). This story fixes *authorization at provisioning* (what they get once in). Both are needed: even with real JWT verification, a legitimately-authenticated-but-unknown ExpTube user would still be auto-ADMIN without this change.

**Affected files:** `src/lib/auth/ensureUserExists.ts` (provisioning logic — change `role: "ADMIN"`→`"USER"` at `:36`, `TenantMember role: "owner"`→`"member"` at `:54`), `src/app/api/workspaces/route.ts` (GET auto-repair create-branch `:18`), and **(only if step-4 gates are added)** the destructive `withTenant` KB routes (pages/blocks/databases update/delete/archive/purge). **No `Prisma` `Role` enum change** (we reuse `USER`, not add `MEMBER`). Tests are **net-new** (no existing `ensureUserExists` coverage — `src/__tests__/lib/auth/` has only `withAdmin.test.ts`).

## Acceptance Criteria
> **CRITICAL CORRECTION (live Codex 2026-06-13): the Prisma `Role` enum is `ADMIN | USER` — there is NO `MEMBER` value.** Confirmed in `prisma/schema.prisma:15-18`, the generated client `src/generated/prisma/enums.ts:12`, the init migration `20260220000000_init/migration.sql:5`, and `tests/api/schema.test.ts:25` (which asserts *only* `ADMIN`/`USER`). The `MEMBER` value the original plan used belongs to the **separate `TeamspaceRole`/`TeamspaceMember` enum** (`OWNER|ADMIN|MEMBER|GUEST`), not `User.role`. **So the schema-compatible least-privilege value is `role: "USER"`, not `"MEMBER"`.** Every "MEMBER" below for `User.role` is corrected to `USER`. (`TenantMember.role` is a free-text `String` default `"owner"` — `schema.prisma:138` — so `"member"` *is* valid there and stays.) Adding a real `MEMBER` to `Role` would be an `ALTER TYPE` migration + regenerated client + validator/test updates — explicitly NOT done here; we reuse the existing `USER`.

**(Default posture — preserves the shared-KB demo, closes the ADMIN hole)**
- [ ] A brand-new SSO user is provisioned with **`role: "USER"`** (the existing Prisma `Role` enum value; **NOT `ADMIN`**, and **NOT `"MEMBER"` which does not exist in `Role`**), and the corresponding `TenantMember.role` is `"member"` (free-text), **not `"owner"`**. S4's "every new user is ADMIN/owner" no longer reproduces.
- [ ] A `USER`-role user can **read** the shared chemistry KB (so the voice agent / `kb-query` and the UI still work for them) but **cannot perform destructive/admin actions**. **(live Codex MUST-FIX — the downgrade is mostly a NO-OP today):** `withAdmin` (`withAdmin.ts:43` enforces `role !== "ADMIN"`) is wired to **only** user-management (`users/route.ts:54`, `users/[id]/route.ts:43`) + the broadcast inline check (`broadcast/route.ts:17`). **The destructive KB routes are `withTenant`, which does NOT check `User.role`:** page update/delete (`pages/[id]/route.ts:79,277`), block update/delete (`blocks/[id]/route.ts:44,101`), database update/delete (`databases/[id]/route.ts:60,210`), archive (`pages/[id]/archive/route.ts:12`), purge (`pages/[id]/purge/route.ts:14`). So flipping new users to `USER` does **not** stop them deleting shared KB content unless this story **adds role gates on those routes**. This AC is therefore load-bearing: either add `ADMIN`/owner gates to the destructive KB routes, or accept that S4's write/delete concern is only partly closed and scope it explicitly. *(Verified against source.)*
- [ ] **(MUST-FIX, Gemini — scope CORRECTED by live Codex) The `workspaces/route.ts` `TenantMember` auto-repair is reconciled.** **Correction:** the upsert at `:13-21` uses `update: {}`, so it does **NOT** re-promote an *existing* `member` to `owner` on a normal `/api/workspaces` GET — the "silently re-promotes on the next call" framing was overstated. The real exposure is narrower but still real: the **create branch** (`:18`) mints `role: "owner"` whenever **no** `TenantMember` row exists (a partial-provisioning failure, a legacy user, or a personal/default-tenant divergence) — so the GET auto-repair path needs a least-privilege policy (create as `"member"`, reserve `"owner"` for genuine tenant creators). The POST-creates-a-tenant path (`:80`) correctly keeps `"owner"` (the user created that tenant). Reconcile the **GET auto-repair create** to `"member"`; leave the POST tenant-creation `"owner"` as-is. *(Verified: `update:{}` confirmed at `route.ts:17`.)*
- [ ] **(live Codex) Agent write/delete paths bypass `User.role` entirely** — `withAgentAuth` authorizes by **scopes**, not role (`agent/auth.ts:61` mock gets `["read","write"]`; `:147` API-key auth returns `["read","write"]` without consulting `User.role`). So agent page create/update/delete (`agent/pages/route.ts:204`, `agent/pages/[id]/route.ts:20,184`) stay writable regardless of the `USER` downgrade. The "least-privilege user cannot do destructive actions" invariant must explicitly **exclude the agent path** (it's scope-governed, and the mock-scope hole is Story 01's job), or state that limiting agent writes requires scope-narrowing, not a role flip.
- [ ] Existing users (already in the default tenant) are **unaffected** by default — the change applies to *new* provisioning only; no migration downgrades current ADMINs unless explicitly opted into (see Risks).
- [ ] The `P2002` unique-constraint recovery path (`:62-81`) still returns the existing user unchanged.
- [ ] The live demo is **not** broken: a newly-provisioned user, hitting `kb-query` (directly or via the Gateway with their identity), still sees the seeded/synced chemistry KB because they are a member of the shared tenant (`User.role: "USER"`, `TenantMember.role: "member"`) — they are **not** siloed into an empty personal tenant. (Confirmed `kbQuery.ts` filters strictly by `tenantId` with no cross-tenant read-through, so staying in `DEFAULT_TENANT_ID` is what preserves retrieval.)

**(Opt-in personal-tenant posture — future multi-tenant; OFF by default)**
- [ ] When `SKB_PERSONAL_TENANT_BY_DEFAULT=1` (or equivalent flag), a new user with no `user_metadata.tenantId` and not on an allowlist gets a fresh personal tenant instead of the shared one. This is documented as breaking the shared-KB demo for those users and is off by default.

- [ ] Tests cover: new user → **`USER`** + `TenantMember member` (shared tenant) by default; workspaces GET auto-repair creates `member` (not `owner`); existing user → unchanged; `P2002` path → existing returned; (if implemented) personal-tenant flag → fresh tenant; (if step-4 gates added) destructive-route 403 for `USER`. `npm test` green — **and** the pre-existing `tests/api/schema.test.ts:25` (asserts `Role` = `ADMIN`/`USER`) still passes because we did NOT add a `MEMBER` enum value.

## Implementation Plan
1. **Default policy = least-privilege `USER` in the shared tenant (REVISED per review; role value corrected by live Codex).** Keep new users in `DEFAULT_TENANT_ID` (so the shared KB / voice agent keeps working) but provision them as **`role: "USER"`** (the existing enum value — NOT `"MEMBER"`, which is not in `Role`) + `TenantMember.role: "member"`. This is the minimal change that closes S4 without breaking the single-shared-KB product and **without** an enum migration. The personal-tenant path becomes an explicit opt-in flag, not the default.
2. **Change the role, not (by default) the tenant, in `ensureUserExists.ts:31-56`.** Set **`role: "USER"`** at `:36` (currently hardcoded `"ADMIN"` — confirmed) and `TenantMember … role: "member"` at `:54` (the upsert, currently `"owner"` — confirmed). Keep the `resolveDefaultTenant()` resolution (`:24-28`) so users still land in the shared tenant. Only when `SKB_PERSONAL_TENANT_BY_DEFAULT=1` do you swap to `createPersonalTenant` (`:97-103`, confirmed) — and then the user is `owner` of *their own* tenant, which is fine. **Do NOT introduce `role: "MEMBER"` on `User`** — it would require an `ALTER TYPE Role ADD VALUE 'MEMBER'` migration, a regenerated Prisma client, and updates to `src/lib/validation/users.ts:18` + `tests/api/schema.test.ts:25` (which asserts only `ADMIN`/`USER`). Out of scope; reuse `USER`.
3. **(MUST-FIX — scope corrected by live Codex) Reconcile only the workspaces-route GET auto-repair create-branch.** `workspaces/route.ts:13-21` upserts with `update: {}` (so it does **not** overwrite an existing `member` — the re-promotion fear was overstated) but the **create** branch (`:18`) mints `"owner"` when no membership row exists. Change the GET auto-repair create to `role: "member"` (least-privilege for an auto-repaired membership). **Leave `:80` (`POST` tenant creation) as `"owner"`** — that user genuinely created the tenant. Verification: a test that a user with no `TenantMember` row hitting `GET /api/workspaces` is auto-repaired as `member`, not `owner`; and that creating a workspace via `POST` still makes them `owner`.
4. **Enforce the role where it matters — REQUIRES NEW GATES (live Codex: today it's a no-op).** The destructive shared-KB routes are `withTenant`, which does **not** consult `User.role`: page update/delete (`pages/[id]/route.ts:79,277`), block update/delete (`blocks/[id]/route.ts:44,101`), database update/delete (`databases/[id]/route.ts:60,210`), archive (`pages/[id]/archive/route.ts:12`), purge (`pages/[id]/purge/route.ts:14`). So flipping new users to `USER` changes nothing for these unless this story **adds explicit `ADMIN`/owner gates** (e.g. a `requireAdmin(ctx)` check, or migrate them to `withAdmin`, or a `TenantMember.role`-based check). **`NEEDS USER INPUT:` should a non-owner `USER` be blocked from delete/archive/purge of shared KB content?** If yes, this story adds gates on the routes above (scope creep beyond `ensureUserExists` — confirm appetite). If no, the story closes only the *auto-ADMIN-on-user-management* surface (`withAdmin` routes) and S4's write/delete-of-shared-content concern is explicitly deferred. Verification (if gates added): a `USER` is 403 on delete/archive/purge, 200 on read. **Note:** the **agent** write path is scope-governed (step's AC above) and is NOT covered by any role gate — out of scope here.
5. **Personal-tenant opt-in branch.** Behind `SKB_PERSONAL_TENANT_BY_DEFAULT`, route new users to `createPersonalTenant`; document it breaks shared-KB visibility for those users. Verification: flag-on test → fresh tenant.
6. **Preserve the email-collision recovery (`:62-81`).** Leave the `P2002` handler intact; orthogonal to the role change.
7. **Tests — all NET-NEW (live Codex: no existing `ensureUserExists` coverage).** There is **no** current `ensureUserExists` test (the only auth test in `src/__tests__/lib/auth/` is `withAdmin.test.ts`; zero references to `ensureUserExists` in `src/__tests__`/`tests` — confirmed). So these are new files, not extensions: provisioning as **`USER`** + `TenantMember member` by default; the `P2002` recovery path returns the existing user unchanged; the workspaces GET auto-repair creates `member` (not `owner`); the opt-in personal-tenant flag; and (if step 4 gates are added) the destructive-route 403s. Also update any test that assumes new users are ADMIN. Do NOT reference a nonexistent prior `ensureUserExists` test as the baseline.

## Risks & Open Questions
- **THE central decision — shared-KB vs isolation (Gemini-surfaced).** The product today is a single shared KB; `kb-query` is tenant-scoped with no read-through (`kbQuery.ts:159,165,193`), and sync writes to `DEFAULT_TENANT_ID`. So "personal tenant per user" silos every new user away from all seeded/synced knowledge and breaks the live voice demo. `NEEDS USER INPUT:` Confirm the intended model: **(default, recommended) everyone stays in the one shared KB but new users are MEMBER not ADMIN** (closes the auto-ADMIN hole, demo keeps working); OR **(opt-in) true per-user personal tenants** (multi-tenant isolation, but the shared chemistry KB is then invisible to new users unless a read-through/overlay is built — which is a separate feature). Assumption taken: **MEMBER-in-shared-tenant by default**; personal-tenant behind a flag.
- **(MUST-FIX) `workspaces/route.ts` re-promotion:** the `TenantMember` auto-upsert to `role: "owner"` (`:13-21`, `:80`) will undo the MEMBER provisioning on the next `/api/workspaces` call unless reconciled (Implementation step 3). This is the subtle bug that would silently re-open S4.
- **MEMBER users + admin-only UI (Gemini):** if `WorkspaceDropdown`/settings/invite UI shows ADMIN-only actions to MEMBERs, they'll hit 403s. Either hide those controls for MEMBERs or accept the 403. Confirm the UI degrades gracefully. Out-of-scope to redesign the UI; in-scope to ensure the *API* enforces ADMIN (step 4).
- **Existing over-privileged users:** current users are already ADMIN/owner in the default tenant. This story does not downgrade them by default (avoids breaking live access mid-flight). `NEEDS USER INPUT:` do you want a follow-up cleanup that demotes existing non-creator members to MEMBER? Flagged, not implemented here.
- The workspace-switch cookie (`skb_active_workspace`, `tenantContext.ts:96`) already verifies `TenantMember` membership before honoring an override — unaffected by the role change (MEMBERs still have a membership row in the shared tenant).
- Source signup is on ExpTube's Supabase (not closeable from this repo). Closing/confirming ExpTube signup is a separate ecosystem item (ExpTube roadmap #15); this story makes SKB safe (no auto-ADMIN) **regardless** of ExpTube's signup posture — which is the real value, since open signup there can't be fixed here.

## Out of Scope
- Authentication mechanism / JWT verification (Story 01).
- Closing or confirming ExpTube's Supabase signup (ExpTube repo).
- An admin UI for managing invitations (existing `TenantMember` flow is reused; no new UI here).

### Reviewer Feedback / Kimi (runtime) — FALLBACK: self (Opus 4.8); reason: Kimi CLI returned HTTP 429 usage-limit.
Runtime breakage / missed bugs:
1. **Order-of-operations race in `ensureUserExists`:** `prisma.user.create` (`:31`) then `prisma.tenantMember.upsert` (`:46`) are two awaits, not a transaction. If two requests for the same brand-new user race (e.g. parallel tab loads), both may pass the `findUnique` (`:15`) miss, then one hits `P2002` on `user.create` (handled at `:62-81`) but the membership upsert could create duplicate-ish state. Low-probability; the `P2002` handler covers the user row but not a racing membership. Note: wrap user+member creation in `prisma.$transaction` for atomicity. Folded as a hardening note.
2. **The role value flows into `TenantContext.role` and is used for authorization downstream** — so flipping new users to MEMBER is only *effective* if some route actually checks `role === "ADMIN"`. If no route enforces it today, MEMBER and ADMIN are functionally identical and S4 is only cosmetically fixed. Step 4 (enforce the role on destructive ops) is therefore load-bearing, not optional — emphasized.
3. **`createPersonalTenant` (`:97-103`) creates a tenant with no seeded content and no `TenantMember` for the user in the *default* tenant** — confirming the silo. Under the opt-in flag this is expected; by default we don't take this path. Confirmed.
4. **No ID collision risk:** `User.id` = `supabaseUser.id` (`:33`), globally unique from Supabase; personal-tenant ids are `prisma.tenant.create` UUIDs. No collision. Confirmed.
The runtime-critical insight is item 2: the role change must be backed by an actual enforcement check or it's a no-op. Folded into AC + step 4.
