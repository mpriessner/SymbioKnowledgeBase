## Problem / motivation

Owner decision #4 (see epic brief): printed document QR codes must resolve to an SKB page URL via the existing publish-to-web feature. SKB already has everything needed to publish a page publicly — `PublicShareLink` model (`prisma/schema.prisma:409-432`, fields `token`, `expiresAt` at 10 years, `revokedAt`, `allowIndexing`), and the full CRUD surface at `GET|POST|DELETE|PATCH /api/pages/[id]/publish` (`src/app/api/pages/[id]/publish/route.ts`) which mints a 24-char token (`crypto.randomUUID().replace(/-/g, "").slice(0, 24)`) and serves it at `${baseUrl}/shared/${token}` via `src/app/shared/[token]/page.tsx` (see `SharedPageContent.tsx`). What's entirely missing is: (a) turning that URL into an actual QR code image, and (b) a way to print one or many of them on paper next to the physical items they represent (reagent bottles, protocol binders, equipment). This is greenfield — grep confirms no QR *generation* code exists anywhere in SKB today (only recognition/classification exists in other repos, see a71-10).

## Proposed change

### 1. QR image generation (client-side, no external service)
Add a QR-encoding library dependency (e.g. `qrcode` npm package — pure client/server JS, no network call, no external API key). Given a share URL string, render an SVG or PNG QR code in the browser. Do not build a server-side QR image endpoint unless printing requires it (a client-rendered `<canvas>`/SVG is sufficient for both on-screen display and browser print-to-PDF).

**Import shape (Round 2):** a default/whole-module import of `qrcode` can pull Node's `canvas` polyfill paths into the client bundle. Import the specific function (`import QRCode from "qrcode"`, then call `QRCode.toDataURL`/`toCanvas`) rather than a `require`-style server-only path, gate all QR rendering behind `"use client"`, and ensure the print route's server shell (if any) never imports `qrcode` at module top level or SSR will try to resolve `canvas`.

### 2. Per-page "Get QR" UI
On any page's action menu (works for document pages from a71-08, but also experiment pages, protocol pages, etc. — this is general infrastructure, not document-specific):
1. If the page is not yet published, call `POST /api/pages/[id]/publish` (existing route, unmodified) to mint the share link. **Guard (Round 1, BLOCKER fix):** the publish route publishes any page regardless of `spaceType`, and `/shared/{token}` is fully unauthenticated — so if the target page is `PRIVATE` or in a restricted teamspace, this step must first show an explicit "this will make the page publicly viewable by anyone with the link" confirmation. Never auto-publish a private page silently; a plain shared-teamspace page can publish without the extra prompt. **Idempotency guard (Round 2 — required):** the route's existing/create check (`findFirst` → `create` if null) is only idempotent for *sequential* calls — two concurrent POSTs for the same page both see no existing link and both mint distinct tokens (the `@unique` constraint is on `token` only, not `(page_id) WHERE revoked_at IS NULL`), leaving two active links for one page. `QrPanel` MUST disable the "Get QR" button while the POST is in flight (mirror `ShareDialogPublish`'s existing `isPending` pattern) to prevent double-click/two-tab double-mint; a DB-level fix (partial unique index on `(page_id) WHERE revoked_at IS NULL`) is the more robust option and recommended if double-mint is still observed under this guard. Reuse the existing `usePublishPage` hook's `res.ok` check rather than hand-rolling a `=== 200` comparison — the route returns **201** on first publish and **200** on the idempotent re-publish, and a hand-rolled `=== 200` check would misclassify first-publish as a failure.
2. **QR payload (Round 2 — corrected):** the QR does NOT encode the publish response's `url` field verbatim — that field is server-derived from `x-forwarded-host`/request origin (`http://localhost:3000` in dev, proxy-dependent in prod) and exists for the legacy copy-link UI only. Instead, reconstruct the payload client-side as `${NEXT_PUBLIC_PUBLIC_BASE_URL}/shared/${response.data.share_token}` (see Section 5's canonical-base-URL env var), and run the localhost/loopback guard (AC9) against this reconstructed string, not the response's `url`.
3. Show it inline in a panel: QR image + page title + one-liner (`Page.oneLiner` if set, else a truncated summary).
4. "Print QR" action opens a dedicated print view. **Render-before-print (Round 2):** `QRCode.toCanvas`/`toDataURL` is async — `await Promise.all([...])` every QR render before enabling the Print button or calling `window.print()`; auto-printing on mount or before render completes produces a blank/partial code that is silently unscannable (AC3/AC4's "no clipped QR" only checks geometry, not whether the code was actually drawn).

### 3. Printable sheet (single + batch)
New route `src/app/print/qr-sheet/page.tsx`. **Component type (Round 2 — required):** this route needs client-side state (the selected page IDs) and must call `window.print()`, neither of which a default server component in the App Router can do — implement it as a `"use client"` page (or a thin server shell wrapping a client child), not a plain server component.
- **Data-passing contract (Round 2 — required):** selected page IDs ride in via query string, `?ids=u1,u2,...` (UUIDs, comma-separated). URL-length limits (~2k conservative, ~8k in Chrome) bound a single print request to roughly 50–200 page IDs; validate the count and reject anything beyond that cap with a clear message rather than silently truncating the batch (AC4's "no overlap" must not silently drop pages past the cap).
- Single mode: one QR + title + one-liner, sized to fit a label or a quarter of an A4 sheet.
- Batch mode: user multi-selects several pages (e.g. from a page-tree checkbox mode or a "Documents" list view built in a71-08/a71-11), and the print view lays out N QR codes on one or more A4 sheets. **Layout decision (Round 2 — pinned, not left open):** default to a full A4 grid for v1, not an Avery 5160 label layout — Avery 5160 is US Letter stock (8.5×11in, 3×10 labels) and printing A4-sized `@page` CSS onto it misaligns every label. State the `@page` size and margins explicitly in the implementation.
- **Batch + private-page confirmation policy (Round 2 — required, resolves an AC4/AC8 conflict):** AC8 requires an explicit per-page confirmation before publishing any PRIVATE/restricted page, but batch mode selects N pages at once with no defined interaction between the two. Naively this either fires N silent POSTs (violates AC8), pops one confirmation per private page (unusable for a 12-up sheet), or aborts the whole batch on any one private page (can't print a mixed sheet). **Policy:** pre-flight the selection into three groups — already published / shared-teamspace (auto-publishable) / private (needs confirmation) — show *one* grouped confirmation listing the titles of the private pages in the batch, and publish per group only after that single confirmation is accepted.
- **Partial-failure handling (Round 2 — required):** batch mode is N+1 HTTP round-trips (one publish call per unpublished page) with no server-side batch endpoint. Publish sequentially per page, accumulate errors, and show a "X of Y published, these failed: …" summary before opening the print view — do not fire all POSTs in parallel with no rollback/error surface.
- Print CSS uses `@media print` rules; no server round-trip needed since the QR data (all published URLs) can be fetched once and rendered client-side before printing.

### 4. Revoke handling
The publish route already supports `DELETE /api/pages/[id]/publish` (revokes all active links). Once revoked, an existing physical QR sticker printed earlier becomes dead: `src/app/shared/[token]/page.tsx` **already** treats a revoked or expired token as not-found — `getShareLink` returns `null` when `revokedAt !== null` or `expiresAt < now`, and the page calls `notFound()` (verified in Round 1). So a scanned dead QR yields a clean 404, not a leak or a hang; no separate bug to file here. Document this tradeoff plainly in the UI: revoking a page's publish link invalidates every physical QR printed against it, and re-publishing mints a *new* token — printed QRs cannot be "reactivated," only reprinted. Add a confirmation dialog on revoke that says exactly this if any QR sheet has ever been printed for that page (track a simple `qrPrintedAt` timestamp, or skip the tracking and just always show the warning — simpler, recommend this for v1).

### 5. Canonical, scannable base URL (Round 1; naming corrected Round 2)
The publish route derives the share URL host from `x-forwarded-host` or `new URL(req.url).origin`, which is `http://localhost:3000` in local/dev and proxy-dependent in prod. A QR encoding `localhost` is unscannable from a phone, and the host must match a71-10's `SKB_SHARE_HOST_ALLOWLIST` or recognition silently fails. Introduce/require an env-driven canonical public base URL for QR rendering, and **fail loud** — refuse to render or print a QR (with a clear message) when the resolved host is `localhost`/loopback, rather than emitting an unscannable code.

**Naming (Round 2 — supersedes Round 1's bare `PUBLIC_BASE_URL`):** because `QrPanel` is a client component, the variable MUST be prefixed `NEXT_PUBLIC_` — a bare `PUBLIC_BASE_URL` is stripped at build time and reads as `undefined` in the browser, since Next.js only exposes `NEXT_PUBLIC_`-prefixed vars client-side. Name it `NEXT_PUBLIC_PUBLIC_BASE_URL`, and register it in `src/lib/env.ts` via `requireInProduction(...)` (warn-in-dev / throw-in-prod), matching the existing `NEXT_PUBLIC_SUPABASE_URL`/`_ANON_KEY` pattern. `QrPanel` reads `process.env.NEXT_PUBLIC_PUBLIC_BASE_URL` directly and uses it to reconstruct the QR payload per Section 2 step 2 — it never consults the publish response's `url` field for QR rendering.

### 6. No *new* publish-route auth surface (but a new exposure vector — see finding 1)
The QR encodes the same `/shared/{token}` URL the publish feature already serves unauthenticated. The publish *route itself* is reused unmodified. However, this story DOES introduce a new exposure vector by surfacing one-click publish on pages a human might otherwise never publish — see Reviewer feedback finding 1 and the section-2 private-page confirmation guard. `SharedPageContent.tsx` was confirmed not to leak sibling/parent links (read-only single-page view). The security review must treat the private-page publish confirmation as in-scope, not defer to "the publish feature is unchanged."

## Acceptance criteria
1. From any page's action menu, "Get QR" publishes the page (if not already published) and displays a scannable QR code that encodes exactly `${baseUrl}/shared/${token}`.
2. Scanning the rendered QR with a phone camera navigates to the correct `/shared/{token}` page and shows the expected content.
3. "Print QR" opens a print-ready view with the QR, title, and one-liner, and produces a correctly laid-out page via the browser's print-to-PDF (verify visually, no clipped QR, readable at printed scanning distance — test print at actual A4 scale, not just full-bleed on screen).
4. Batch mode: selecting 3+ pages produces a single print view with all QR codes laid out on one or more A4 sheets (not Avery label stock for v1 — see Section 3), no overlap, each labeled with its own title.
5. Revoking a page's publish link (existing `DELETE /api/pages/[id]/publish`) is preceded by a warning that any previously printed QR for that page will stop working, with no code changes to the revoke logic itself.
6. Re-publishing after a revoke mints a new token (existing behavior, confirm still true) and requires reprinting — the UI must not claim the old sticker will start working again.
7. No new unauthenticated data exposure: the `/shared/{token}` page shown after a QR scan exposes nothing beyond what today's publish feature already exposes for that token.
8. Generating a QR for a `PRIVATE` (or restricted-teamspace) page first requires an explicit "this makes the page publicly viewable by anyone with the link" confirmation; the page is never auto-published silently by the "Get QR" action.
9. When the resolved base-URL host is `localhost`/loopback, the "Get QR"/print flow refuses with a clear message instead of rendering an unscannable QR; a configured canonical public host (`NEXT_PUBLIC_PUBLIC_BASE_URL`) produces a QR that a phone camera resolves to the correct `/shared/{token}` page. The QR payload is reconstructed client-side from `share_token` + this canonical base, not read from the publish response's `url` field.
10. Rapid double-click or two-tab access to "Get QR" on the same unpublished page does not mint two active share links for that page — the publish action is disabled client-side while in flight, and if a DB-level fix (partial unique index) is added, it is enforced there too.
11. A batch selection containing a mix of private and already-shareable pages shows exactly one grouped confirmation listing the private page titles before publishing proceeds — it does not pop one dialog per private page, and does not silently skip the confirmation for any private page in the batch.
12. Selecting more page IDs than the print route's supported count (per the `?ids=` length cap) is rejected with a clear message, not silently truncated.
13. The print view does not enable printing (or auto-print) until every QR in the batch has finished rendering — no blank/partial codes reach the printed page.
14. If publishing fails for some pages in a batch, the user sees which pages failed before the print view opens, rather than an incomplete sheet with no explanation.

## Regression risks
- **New dependency (`qrcode` or similar)**: pure client-side, should not affect server bundle size materially — verify build output doesn't balloon the client bundle if the library is large; prefer a package with a small dedicated SVG-rendering path over one that pulls in canvas polyfills server-side. **Import shape (Round 2):** import the named function (`QRCode.toDataURL`/`toCanvas`), not a default/whole-module import — the latter can pull Node's `canvas` polyfill into the client bundle and break SSR if imported at a server module's top level.
- **Publish route reuse**: calling `POST /api/pages/[id]/publish` from a new UI surface must not change its idempotency behavior for *sequential* calls (existing behavior: repeated POSTs on an already-published page return the same active link, not a new one) — do not accidentally add force-remint logic that would invalidate previously printed QRs on every "Get QR" click. **Concurrent calls are NOT idempotent today (Round 2):** two simultaneous POSTs for the same page can both mint distinct tokens (no partial-unique-index enforcement on `(page_id) WHERE revoked_at IS NULL`); this story is the first caller that makes concurrent access to this route trivial (double-click, two tabs), so the client-side in-flight guard in Section 2 is required, not optional.
- **Print CSS bleed into normal UI**: `@media print` rules must be scoped tightly (e.g. under a dedicated print-only route/component) so they don't leak into the regular page-editing UI's screen styles.
- **Batch selection UI collision**: if a71-11's "documents list" view is built concurrently, coordinate the multi-select component so this story and that one don't both build competing selection UIs — check in with whichever session picks up a71-11 before implementing batch selection here.

### Round 2 — GLM-5.2 runtime lens (2026-07-04)

1. **BLOCKER — `PUBLIC_BASE_URL` (Round 1's fix) is unreachable from the client component that renders the QR, so AC9 cannot work as written.** Section 1 mandates a *client-side* QR library rendered in the browser, and section 5 introduces `PUBLIC_BASE_URL` as the canonical host. Next.js only exposes env vars to client components if they are prefixed `NEXT_PUBLIC_`; a bare `PUBLIC_BASE_URL` is stripped at build time and reads as `undefined` in `QrPanel.tsx`. At runtime the QrPanel then either silently falls back to `window.location.origin` (≈ `localhost:3000` in dev → unscannable QR, the exact failure AC9 promises won't happen) or throws. The repo already uses the `NEXT_PUBLIC_` convention (`NEXT_PUBLIC_SUPABASE_URL`, `_ANON_KEY`), and `src/lib/env.ts` is the fail-fast gate that must register the new var. **Fix:** name it `NEXT_PUBLIC_PUBLIC_BASE_URL`, add it to `src/lib/env.ts` via `requireInProduction(...)` (warn-in-dev / throw-in-prod, matching the Supabase vars), and have the QrPanel read `process.env.NEXT_PUBLIC_PUBLIC_BASE_URL`.

2. **BLOCKER — the story contradicts itself on which string the QR actually encodes, and the literal reading re-introduces the localhost bug Round 1 tried to kill.** Section 2 step 2 says "Render the QR code encoding the exact string `${baseUrl}/shared/${token}` **returned by that call**" — i.e., use `response.data.url`. But `publish/route.ts` lines 40-42/83-85/112-114 derive that `url` from `x-forwarded-host` / `new URL(req.url).origin`, which is `http://localhost:3000` in dev and proxy-dependent in prod. Section 5's `PUBLIC_BASE_URL` canonicalization only has any effect if the client *reconstructs* the URL from `share_token` + the canonical base and **ignores** the server's `url` field. The story never says to do that, so an implementer following section 2 literally will encode `response.data.url` verbatim — and the AC9 localhost guard (which checks the *resolved* host) will fire every time in dev with no way to override it, because the override var is never consulted. **Fix:** state explicitly that the QR payload is `${NEXT_PUBLIC_PUBLIC_BASE_URL}/shared/${response.data.share_token}`, reconstructed client-side; the server's `.url` field is for the legacy copy-link UI only and must not drive QR rendering. The localhost guard then runs against this reconstructed string.

3. **MAJOR — batch mode + AC8's private-page confirmation are uncomposed; the batch flow can deadlock or silently violate AC8.** AC8 requires an explicit per-page "this makes the page public" confirmation before publishing any PRIVATE/restricted page. Batch mode (section 3) multi-selects N pages and prints them in one shot — but says nothing about how the confirmation applies to a mixed batch. At runtime one of three things happens, all bad: (a) the batch fires N POSTs with no prompt → silently violates AC8 for every private page in the selection; (b) the batch pops one confirmation per private page → 12 modal stacking for a 12-up sheet, unusable; (c) any private page aborts the whole batch → user can't print a sheet that includes even one private doc. **Fix:** specify the policy — recommended: pre-flight the selection, partition into "already published" / "shared-teamspace, auto-publishable" / "private, needs confirmation," show *one* grouped confirmation listing the private page titles, and publish/abort atomically per group. Without this, AC4 and AC8 are mutually unsatisfiable.

4. **MAJOR — the print route's data-passing mechanism is unspecified and the obvious choice has hard runtime limits.** `src/app/print/qr-sheet/page.tsx` is an App Router page (server component by default), but the flow needs (a) the selected page IDs from the listing view and (b) `window.print()` (client-only). Server components can't read react/localStorage state and can't call `window.print()`; so the page must be `"use client"` or a thin server shell wrapping a client child. The IDs have to ride in via query string (`?ids=u1,u2,...`) — UUIDs are 36 chars each, so URL-length caps (~2k conservative / ~8k Chrome) bound a batch to roughly 50–200 pages. That's fine for v1 but unbounded multi-select on a large documents list will hit it. **Fix:** declare the print page a client component (or hybrid), document `?ids=` as the contract, and cap/validate the ID count (reject with a clear message beyond N) so AC4's "no overlap" doesn't silently truncate.

5. **MAJOR — the POST idempotency claim in "Regression risks" is only true for *sequential* calls; concurrent calls double-mint, and this story is the first caller that makes concurrency trivial.** `publish/route.ts` lines 73-93 do `findFirst(existing)` → if null, `create`. Two concurrent POSTs for the same `pageId` both see `existing === null` and both `create` distinct tokens (the `@unique` is on `token` only, not on `(pageId WHERE revokedAt IS NULL)`). Result: two active, non-revoked links for one page; later GET/POST returns whichever `findFirst` happens to return first (no `orderBy` on the POST existing-check, so non-deterministic); a subsequent DELETE revokes *both*. The regression note's promise ("repeated POSTs return the same active link") breaks under double-click or two-tab access. **Fix options (pick one):** add a partial unique index on `(page_id) WHERE revoked_at IS NULL` (cleanest, DB-enforced), or use `upsert`, or — minimum for this story — disable the "Get QR" button while the POST is in flight (the existing `ShareDialogPublish` does this via `isPending`; the new `QrPanel` must too) and debounce batch POSTs to be sequential-per-page.

6. **MINOR — POST returns **201** on first publish and **200** on the idempotent re-publish; the QrPanel must not special-case 200.** Confirmed at `publish/route.ts:124` (201) vs `:87` (200). The existing `usePublishPage` hook correctly uses `res.ok`, so reuse it verbatim; if the new code hand-rolls `fetch` and checks `=== 200`, the first-publish path is misclassified as an error and no QR renders. Worth a one-line note in the story so the implementer doesn't regress.

7. **MINOR — batch mode is N+1 HTTP round-trips with no partial-failure handling.** A 12-up sheet of unpublished pages fires 12 sequential-or-parallel POSTs from the client; a single network blip leaves some pages published and others not, with no rollback and (per finding 3) unclear UI state. Recommend sequential publish with per-page error accumulation and a "X of Y published, these failed: …" summary before the print view opens, or a server-side batch-publish endpoint.

8. **MINOR — async QR render races `window.print()`.** `QRCode.toCanvas`/`toDataURL` is async; if the print view auto-calls `window.print()` on mount (or the user clicks before render finishes), the printed sheet contains blank/partial codes — silently unscannable, and AC3/AC4 ("no clipped QR") only checks *geometry*, not *whether the code was actually drawn*. `await Promise.all([...])` all renders before enabling the Print button / firing `window.print()`.

9. **MINOR — Avery 5160 and A4 are not the same physical sheet; the default-layout spec conflates them.** Section 3 says "default something printable on a standard label sheet like Avery 5160-equivalent or full A4 grid." Avery 5160 is US Letter (8.5×11in, 30 labels, 3×10) — printing A4-sized `@page` CSS onto 5160 stock misaligns every label, and AC3/AC4's "correctly laid out / no overlap" fails on real label stock even though on-screen it looks fine. Pick one (recommend A4 grid for v1 since the rest of the spec is A4) and state the `@page` size + margins explicitly.

10. **MINOR — `qrcode` import shape affects client bundle and SSR.** Default-importing `qrcode` pulls Node `canvas` polyfill paths into the client bundle (the regression note flags size generally, not this). Import the specific function (`import QRCode from "qrcode"` then call `QRCode.toDataURL` is fine; avoid `require("qrcode")` server-only paths), gate it behind `"use client"`, and ensure the print route's server shell doesn't import it at module top-level or SSR will try to resolve `canvas`.
