## Problem / motivation
Every existing seed script in the notebook repo (`scripts/seed_compare_experiments.py` — full notebook+wiki+README generation, supports `--clean`; `seed_demo_experiments.py`; `seed_voice_test_experiment.py`) creates notebook-only data. None of them touch ExpTube or SKB, and there is no equivalent seed for either of those systems at all. That means the entire sync mesh this batch builds ([a71-01](./2026-07-04-a71-01-notebook-joins-sync-mesh.md) through [a71-04](./2026-07-04-a71-04-agent-wiki-overview-okf.md)) has never been exercised with realistic, multi-system data — every acceptance test so far in this batch is necessarily unit/integration-level with mocked hubs. Without a shared seeding story, whoever implements a71-01/02/03/04 has no fast way to answer "does the whole chain actually agree with itself once real data flows through it," and manual testing would mean hand-creating a dozen experiments three times over in three different UIs.

## Proposed change
A new seeding script, modeled on `seed_compare_experiments.py`'s structure but driving the **real sync chain** rather than writing directly to any database — this is the deliberate design choice: seeding through the actual `POST /api/experiments` → lifecycle-sync → content-sync path is what proves the chain works, whereas direct DB writes to ExpTube/SKB would only prove the seed data looks right, not that sync does.

**1. Script location and shape.** `chatbot-notebook-transparency-prototype-ui/scripts/seed_synchronized_experiments.py`, following `seed_compare_experiments.py`'s CLI conventions (`--clean` flag for teardown, similar argument parsing). For each of 8-10 experiments:
- Call the real `POST /api/experiments` endpoint (not `store.create_experiment()` directly) so a71-01's `eln_experiment_id` allocation and lifecycle-sync dispatch actually fire.
- Write a realistic `notebook.ipynb` body and `wiki.md` (with a proper `<!-- EXEC-SUMMARY:START/END -->` block) via the existing notebook-write path (`dispatch_tool`/the wiki endpoints `GET/POST /api/experiments/{id}/wiki`), so a71-02's `on_notebook_write` content-sync hook fires naturally rather than being simulated.
- Attach 1-2 small media stubs per experiment (tiny placeholder images/PDFs) via the experiment's existing upload path, exercising the attachment/mirror machinery on the SKB side once synced.
- Vary the dataset realistically: mix of reaction types (e.g. reduction, oxidation, esterification, cross-coupling — whatever set `setupHierarchy.ts`'s "Reaction Types" page already documents, reuse those exact categories rather than inventing new ones so the seeded data lines up with SKB's existing taxonomy pages), 3-4 distinct fictional researcher names (matching the "Researchers" hierarchy page concept), a spread of statuses including at least 2 archived experiments (to exercise the Experiments/Archive split in both `setupHierarchy.ts` and a71-04's index).

**2. ExpTube-side data — and a hard gating consequence for SKB.** Notebook-originated experiments produce an ExpTube row only once the ExpTube-side origin CHECK-constraint change (flagged in a71-01 — it is a `TEXT + CHECK` column, not an enum) lands. **Critical, verified consequence the earlier draft missed:** the ExpTube hub relays to SKB (`syncCreateToSKB`) *only after* its own `experiments` INSERT succeeds (`app/api/sync/experiments/route.ts` ~L131 then ~L143). If the origin CHECK is un-patched, that INSERT fails the CHECK and the hub returns HTTP 500 **before** relaying — so a notebook-originated create yields **neither an ExpTube row nor a SKB page**. Therefore, if the notebook content path routes through the hub (a71-02 Option A), the origin CHECK fix is a **hard precondition for AC4 (SKB page exists)**, not merely for the ExpTube-row check — the script cannot "degrade gracefully" and still verify SKB pages via the notebook path. This only relaxes if a71-02 chooses Option B (notebook posts content directly to SKB, bypassing the hub): then SKB pages can appear even without the ExpTube fix, but lifecycle create still routes through the hub per a71-01, so the experiment *page* scaffold still depends on it. **Verification-step behavior:** if the ExpTube API is reachable and the origin migration is in place, assert a matching `origin='notebook'` row per seeded experiment; if the migration is absent, the script must report the seed run as **failed/blocked for SKB verification too** (not a silent skip), because the SKB pages won't exist either — a plain "skip" here would hide a real broken chain.

**3. Teardown (`--clean`).** Must tear down through the **real delete paths**, not direct file/DB deletion: `DELETE /api/experiments/{id}` per seeded experiment (firing a71-01's delete lifecycle sync), confirming the corresponding SKB page moves to Archive (per the existing delete/archive semantics in `/api/sync/experiments`) or is fully purged, matching whatever the `purge` action's real behavior is. Track seeded experiment IDs in a small manifest file (`scripts/.seed_synchronized_manifest.json`) written on creation and consumed on `--clean`, so a partial or interrupted seed run can still be torn down cleanly. Write the manifest using atomic tmp-file-plus-rename semantics on every append (not an in-place edit of the live file), and validate/skip any unparseable trailing entry at load time — a script killed mid-write (crash, `Ctrl+C`) must never leave the manifest in a state that silently loses track of an already-created experiment.

**0. Pre-flight check (run before creating anything).** Before seeding, verify the required dependencies are actually in place: the notebook (`:8000`), ExpTube, and SKB (`:3000`) endpoints are reachable; the ExpTube origin CHECK-constraint migration from a71-01 is present (a cheap read-only probe, e.g. attempt a harmless `create` sync call and check it doesn't 500 on the CHECK, or query the constraint definition directly if the script has DB access); required env vars (`SKB_LIVE_KEY`, etc.) are set. Report all of this clearly before seeding starts rather than discovering a missing dependency 5 experiments into an 8-10 experiment run — this is a cheap check that turns "the run failed halfway through for an unclear reason" into "here's exactly what's not configured yet."

**4. Verification checklist (script output, not just side effects).** After seeding, the script prints a comparison table: for each seeded experiment, its `eln_experiment_id`, whether an ExpTube row exists (with origin), whether a corresponding SKB page exists (with the expected sections populated per a71-02's structure), and whether it appears correctly in a71-04's Experiments Index (active vs. archived section). Any mismatch is printed as a clear `MISMATCH:` line, not just a silent pass/fail — this is meant to be read by a human debugging the sync chain, not just a CI gate.

**5. Voice-tool end-to-end check.** As the final step, the script (or an accompanying short manual runbook printed at the end) poses one `kb-query`-style question about a specific seeded experiment (e.g. "What was the yield on the esterification experiment run by [researcher]?") directly against `POST /api/agent/kb-query`, and separately documents how to ask the same question via the companion (`search_knowledge_base`, [a71-05](./2026-07-04-a71-05-companion-kb-search-tool.md)) and Android (`query_knowledge_base`, [a71-06](./2026-07-04-a71-06-android-kb-search-resilient.md)) voice tools, confirming all three return a coherent, correct answer grounded in the seeded content — this is the concrete proof that the whole batch (sync + content + wiki index + voice search) works together, not just in isolation. Since this step is manual (see Out of scope) and depends on services this script doesn't itself start, the script's printed runbook should say plainly which services/sessions the human needs running before they try it (companion dev server, an active Android debug session with SKB credentials configured) rather than silently assuming they're already up — a failure here should read as "voice session wasn't running," not as a mysterious no-answer.

**chatbot-notebook-transparency-prototype-ui:**

- `scripts/.seed_synchronized_manifest.json` — new runtime file (gitignored), tracks seeded IDs for clean teardown.

## Out of scope
- Building the ExpTube-side origin-enum migration itself (a71-01 flags this as a separate coordination item; this story only conditionally verifies it if present).
- Load/performance testing at scale — 8-10 experiments is a realistic smoke-test size, not a stress test (a71-04's regeneration-concurrency risk is worth revisiting separately at higher volumes if that ever becomes a concern).
- Automating the voice-tool checks (point 5) as a CI-runnable test — those two checks are documented as a manual runbook step since they require a live voice session on each platform, not something this script can drive headlessly in this batch.

## Acceptance criteria
1. Running the script with no prior state creates 8-10 experiments through the real `POST /api/experiments` → sync-dispatch chain, each with a valid `eln_experiment_id`, populated `wiki.md` with a well-formed EXEC-SUMMARY block, and 1-2 attached media stubs.
2. The dataset includes at least 2 archived experiments and at least 3 distinct researcher names and at least 3 distinct reaction types drawn from SKB's existing "Reaction Types"/"Researchers" taxonomy.
3. For each seeded experiment where the ExpTube origin CHECK-constraint change is present, an ExpTube row exists with `origin='notebook'` and the correct `eln_experiment_id`; where it is absent, the script reports the run as blocked (see AC4's coupling) rather than a silent success-with-skip.
4. Each seeded experiment produces a corresponding SKB page with `## Executive Summary`/`## Notebook Narrative` sections populated (per a71-02), reachable via `findExperimentByElnId`. **Note the coupling (verified):** when the notebook routes through the ExpTube hub, this page is created by the hub's `syncCreateToSKB` relay, which fires only after a successful ExpTube-row INSERT — so AC4 cannot pass unless AC3's ExpTube origin CHECK fix is present. The script must surface this as a single coherent "chain blocked at ExpTube" message, not an ExpTube-skip plus a separate confusing SKB-missing error.
5. The a71-04 Experiments Index correctly lists all active seeded experiments in its main section and all archived ones in the Archive section, with the seeded titles/one-liners matching.
6. `--clean` tears down every seeded experiment through the real delete path, leaving no orphaned SKB pages, ExpTube rows, or notebook folders; a second `--clean` run on an already-clean state is a safe no-op.
7. The verification-table output correctly flags at least one deliberately-introduced mismatch when tested (e.g. temporarily skip one experiment's content sync in a test run and confirm the script's `MISMATCH:` line catches it) — this proves the checker itself is trustworthy, not just that happy-path data lines up.
8. A `kb-query` call about a specific seeded experiment's detail (e.g. yield, researcher, reaction type) returns a factually correct answer grounded in the seeded wiki content.

## Verification plan
- Run the script against a local dev stack with notebook (:8000), ExpTube, and SKB all running; inspect the printed verification table for zero mismatches.
- Deliberately break one hop (e.g. stop the SKB service mid-seed, or seed one experiment without ExpTube reachable) and confirm the script reports the failure clearly rather than crashing or silently succeeding.
- `curl` spot-checks matching a71-01/02/04's own verification plans, e.g.:
  ```bash
  curl http://localhost:8000/api/experiments   # confirm seeded notebook experiments present
  curl -H "Authorization: Bearer $SKB_LIVE_KEY" http://localhost:3000/api/agent/search?q=esterification
  curl -X POST http://localhost:3000/api/agent/kb-query -H "Authorization: Bearer $SKB_LIVE_KEY" \
    -d '{"query":"What was the yield on the esterification experiment?","depth":"medium"}'
  ```
- Manual: ask the same question via the companion voice session and via the Android app, confirm both surfaces resolve correctly against the seeded data (this is the final human sign-off step for the whole Epic A batch, not just this story).
- `--clean` verification: run seed → clean → re-run `seed` immediately after, confirm no ID collisions or leftover-file errors from the previous run.

## Regression risks
- **This script is the first real-traffic load the sync chain has seen.** It is likely to surface bugs in a71-01/02/03/04 that their own unit tests missed (e.g. the concurrency-guard behavior a71-04 flags as a real risk under sync bursts) — that is the intended value of this story, but implementers should expect to file follow-up fixes against the earlier stories rather than treating this script's own logic as the bug source by default.
- **Manifest-file drift** if a seed run is killed mid-way (e.g. process crash) — the manifest may list experiments that were never fully created, or miss ones that were created before the crash. Mitigate: write the manifest incrementally (append per-experiment as it's created, not once at the end) so a partial run's manifest is still accurate enough for `--clean` to find everything it created.
- **Fictional data resembling real lab data** could confuse anyone browsing the KB casually. Mitigate: prefix all seeded titles/researcher names with a clearly synthetic marker (e.g. `[SEED]` or a dedicated tag), consistent with how `seed_compare_experiments.py`/`seed_demo_experiments.py` likely already flag their own output — confirm and match that existing convention rather than inventing a new one.
