# SKB-48.5: FS→DB Sync for Databases

**Story ID:** SKB-48.5
**Epic:** EPIC-48 (Database Mirror — Markdown Filesystem Sync)
**Points:** 8
**Priority:** High
**Status:** Not Started
**Depends on:** SKB-48.2 (deserializer), SKB-48.4 (DB→FS sync must exist for echo prevention)

---

## User Story

As an agent, I want to create and edit databases by writing Markdown files, and have those changes automatically sync to the database, so I can manage structured data through the filesystem.

---

## What This Story Delivers

The FileWatcher extended to detect changes in the `Databases/` directory and propagate them to PostgreSQL. Handles three scenarios:

1. **Edit existing database file** — Changed rows update, new rows create, removed rows delete
2. **Create new database file** — New Database + Page + DbRows created from scratch
3. **Delete database file** — Handled cautiously (see below)

---

## Technical Specification

### FileWatcher Extension

Extend `FileWatcher.ts` to handle files matching `Databases/*.md`:

```typescript
// In the chokidar watcher setup, add handler for Databases/ directory

if (relativePath.startsWith("Databases/")) {
  // Route to database handlers instead of page handlers
  handleDatabaseFileChange(filePath);  // or handleDatabaseFileAdd
}
```

### Handler: Edit Existing Database File

```typescript
async function handleDatabaseFileChange(filePath: string): Promise<void> {
  // 1. Check sync lock — skip if locked
  // 2. Read file content
  // 3. Parse with markdownToDatabase()
  // 4. Look up existing database by ID from frontmatter
  // 5. Diff current DB state against parsed file:
  //    a. Schema changes → update Database.schema
  //    b. Row diffing (see strategy below)
  // 6. Apply changes in Prisma transaction
  // 7. Suppress DB→FS echo via sync lock
}
```

### Row Diffing Strategy

Matching rows between the Markdown table and the database is the hardest part. Strategy:

1. **Primary match: by row ID** — If `.skb-meta.json` stores a row order mapping, use that
2. **Fallback match: by TITLE value** — Match rows by their TITLE column value (unique within a database in practice)
3. **Last resort: by position** — Match by row index in the table

**Diff operations:**
- Row in file AND in DB (matched) → compare properties, update if different
- Row in file but NOT in DB → create new DbRow + auto-create linked Page
- Row in DB but NOT in file → delete DbRow (with confirmation pattern — see deletion safety below)

### Handler: New Database File

```typescript
async function handleDatabaseFileAdd(filePath: string): Promise<void> {
  // 1. Check sync lock — skip if locked
  // 2. Parse with markdownToDatabase()
  // 3. Frontmatter has no "id" → this is a new database
  // 4. Validate schema (at least 1 TITLE column, valid types)
  // 5. Transaction:
  //    a. Create Page (for the database to attach to)
  //    b. Create Database with schema from frontmatter
  //    c. For each table row: create DbRow + auto-create linked Page
  // 6. Rewrite .md file with generated IDs (database id, page_id)
  // 7. Update .skb-meta.json
}
```

### Handler: Delete Database File

Database deletion via filesystem is **dangerous** (could lose data accidentally). Strategy:

- **Do NOT auto-delete** the database from PostgreSQL when the file is removed
- Instead, log a warning: `"Database file removed: {path}. Database preserved in DB. Use API to delete."`
- Update `.skb-meta.json` to remove the entry
- The file can be regenerated by triggering a full sync

### Validation

All row property values must pass validation before writing to DB:
- Reuse validation logic from `src/lib/database/propertyValidators.ts`
- Invalid rows → log error with details, skip that row, continue with valid rows
- Schema changes → validate new schema (must have TITLE column, valid types)

### Transaction Safety

All database mutations must be wrapped in a Prisma transaction:

```typescript
await prisma.$transaction(async (tx) => {
  // Update schema if changed
  // Create new rows
  // Update changed rows
  // Delete removed rows
});
```

If the transaction fails, no partial changes are applied.

## Implementation Notes

- **Row matching is imperfect.** TITLE-based matching works for most cases but breaks if two rows have the same title. For robustness, consider storing row IDs as HTML comments in the Markdown table: `| Build feature <!-- row:uuid --> | ...`. This is invisible to humans but gives the sync a reliable anchor. However, this adds complexity — start with TITLE matching and add ID comments as a follow-up if needed.
- **New column in schema:** If an agent adds a column to the frontmatter schema, existing rows in the table won't have values for it. The sync should treat missing columns as null, not error.
- **Removed column from schema:** If an agent removes a column from frontmatter, the table may still have that column's data. The sync should ignore table columns that aren't in the schema.
- **Debouncing:** When an agent writes many changes in quick succession (e.g., a script), debounce to a single sync. Reuse the existing `FS_DEBOUNCE_MS` from config.
