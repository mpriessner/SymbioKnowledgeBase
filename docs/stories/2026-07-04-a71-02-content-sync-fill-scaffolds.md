## Problem / motivation
SKB's sync receiver (`POST /api/sync/experiments`, `src/app/api/sync/experiments/route.ts` — note: the earlier draft's `src/app/app/api/...` path had a duplicated `app` segment and is wrong) is deliberately minimal: a `create` action scaffolds a near-empty templated page under Chemistry KB → Experiments — the page body is generated by a local `generateExperimentKbMarkdown()` helper (verified `route.ts:181`), converted via `markdownToTiptap`, and stored as a single `Block` of `type: "DOCUMENT"` — and `update` only renames the page title. The Zod schema (`syncPayloadSchema`) accepts `fields: z.record(z.string(), z.string()).optional()` — flat string key/value pairs only, no structured body content. Nothing about the actual experiment write-up ever reaches SKB. Meanwhile two rich content sources exist and go nowhere:

- **Notebook `wiki.md`**: an LLM-generated per-experiment summary with a machine-extractable block delimited by `<!-- EXEC-SUMMARY:START -->` / `<!-- EXEC-SUMMARY:END -->`, readable/writable via `GET`/`POST /api/experiments/{id}/wiki`.
- **ExpTube `experiment_analysis`**: `merged_narrative`, `merged_steps`/`materials`/`equipment`/`observations` (JSONB), `report_markdown`, `completeness_score`, with an `is_stale` trigger for staleness tracking — none of this is included in ExpTube's existing `skb-sync-service.ts` sync calls (`syncCreateToSKB`/`syncUpdateToSKB` only pass `title`/`status`-shaped fields per `lib/actions/experiments.ts` L271-491).

Net effect: every SKB experiment page synced from either source is a near-empty stub forever, defeating the point of the agent wiki layer this batch is also building ([a71-04](./2026-07-04-a71-04-agent-wiki-overview-okf.md)) — there's no real content for an agent to read.

## Proposed change
Add a new sync action, `content_update`, to the existing `/api/sync/experiments` endpoint, with **section-owned merge semantics** so human edits made directly in SKB survive being overwritten by the next automated sync.

**1. Payload schema (extends `syncPayloadSchema` in `route.ts`).**
```json
{
  "eln_experiment_id": "EXP-2025-0001",
  "action": "content_update",
  "source": "notebook" | "exptube",
  "correlation_id": "uuid-optional",
  "content": {
    "section": "notebook_wiki" | "exptube_analysis",
    "markdown": "<= 200_000 chars",
    "exec_summary": "<= 4_000 chars, optional — notebook only",
    "completeness_score": 0.0,
    "is_stale": false,
    "generated_at": "2026-07-04T12:00:00Z"
  }
}
```
Add a Zod union member to `syncPayloadSchema`: `action: z.enum([...existing, "content_update"])` plus a conditional `content` object validated only when `action === "content_update"`. Size cap: reject payloads with `content.markdown` over 200,000 characters (return 413) — this is generous but bounded (SKB's `Block.searchVector` FTS indexing and the mirror/attachment quotas already assume bounded page sizes; an unbounded LLM narrative could otherwise blow past reasonable page sizes).

**2. Section ownership map.** An SKB experiment page becomes a page with **named sections** rather than one flat body, using markdown heading anchors the page renderer/editor already understands (Tiptap doc via `markdownToTiptap`, same deserializer the create-path already calls). Fixed section order, each with a clear owner:
```
# <Experiment Title>            (page title — owned by lifecycle sync, a71-01/existing)
## Executive Summary            (owned by: notebook, section key "notebook_wiki" exec_summary)
## Notebook Narrative            (owned by: notebook, section key "notebook_wiki" markdown)
## ExpTube Analysis              (owned by: exptube, section key "exptube_analysis" markdown)
## KB Notes                      (owned by: SKB user edits — never touched by incoming sync; see a71-03)
```
On a `content_update`, the handler parses the existing page's Tiptap doc, locates the heading matching the owned section for `content.section`, and replaces **only that section's subtree** — headings above/below and any content in `## KB Notes` are left untouched. If the section heading doesn't exist yet (first content sync for a freshly-scaffolded page), append it in the fixed order above. This is the concrete mechanism that lets "human edits in other sections survive," per the brief's design goal — it works because sync only ever owns its own named sections, never the whole document.

**3. Idempotency.** Same anchor as the existing lifecycle sync: `Page.externalId` (`eln_experiment_id`) via `findExperimentByElnId`. Add a `generated_at` timestamp comparison: if the incoming `content.generated_at` is older than or equal to the section's last-applied timestamp (stored in a small JSON blob in `Page.metadata` or a new `PageSyncState` model — recommend a lightweight new model `PageSyncSectionState { pageId, sectionKey, lastAppliedAt, lastCorrelationId }` keyed by page + section), skip the write and return `{applied: false, reason: "stale"}` rather than clobbering a newer version with an out-of-order retry (this also plugs a real risk: the notebook's DLQ retry mechanism from a71-01 can deliver events out of order after an outage).

**4. Triggering AND destination (must be decided — the draft was internally contradictory).** The original draft said the notebook reuses `sync_client.py` from a71-01 (which points at the **ExpTube hub**, `NOTEBOOK_SYNC_HUB_URL`) *and* gave a `curl` example POSTing directly to **SKB** (`http://localhost:3000/api/sync/experiments` with SKB's `SYNC_SERVICE_KEY` + `X-Tenant-ID`). Those are two different destinations with two different credentials; the story must pick one:

- **Option A — notebook content goes through the ExpTube hub** (architecturally consistent with a71-01's "notebook is a peer of the hub"): then the ExpTube hub-inbound route (`app/api/sync/experiments/route.ts`) must (i) add `content_update` to its action allowlist — verified it currently rejects any action outside `['create','delete','archive','restore','update','purge']` with HTTP 422 `"Unknown action"` (~L90) — and (ii) relay `content_update` onward to SKB via a new `syncContentToSKB`, while **not** relaying back to the source. This is a **non-trivial, currently-unflagged ExpTube-side change** and must be added to the ExpTube file list below.
- **Option B — notebook content goes directly to SKB** (bypasses the hub, matches the `curl`): then the notebook needs SKB's `SYNC_SERVICE_KEY` + tenant header directly — a *different* credential than a71-01's `NOTEBOOK_SYNC_SERVICE_KEY`-to-hub design — and the ExpTube-analysis content path (which is already ExpTube→SKB) stays symmetric with it. Simpler for content, but splits the notebook's outbound auth across two destinations/keys.

**Recommendation: Option B for content, Option A only if the hub must observe/annotate content.** Content is SKB-page body; the hub adds no value for the notebook→SKB content leg, and Option A forces a hub schema+relay change that Option B avoids. Whichever is chosen, update the credential/config section and the ExpTube file list to match — do not leave both destinations implied.

Notebook trigger mechanism (either option): extend `on_notebook_write` (`backend/tools.py:434`, currently wired to `_bump_reload_required` at `app.py:826`) to also schedule a debounced (e.g. 30s coalescing window) `content_update` push of the current `wiki.md` body + parsed EXEC-SUMMARY block. Note `on_notebook_write` is a synchronous `Callable[[str], None]`; the debounced push must be dispatched onto the event loop (e.g. `asyncio.create_task`) rather than run inline. Because the debounce window is up to 30s, register the pending task with the FastAPI lifespan's shutdown handling (same pattern a71-01 needs for its DLQ drain loop) and flush-or-cancel it on shutdown — otherwise a debounced push scheduled just before the process exits (e.g. `Ctrl+C` right after a save) is silently lost rather than either sent or DLQ'd. ExpTube: hook `experiment_analysis` write paths (wherever `report_markdown`/`merged_narrative` are persisted — the analysis pipeline, ExpTube-side) to call `skb-sync-service.ts` with a new `syncContentToSKB(elnId, {section: "exptube_analysis", markdown: report_markdown, completeness_score, is_stale})` mirroring the existing `syncUpdateToSKB` pattern.

## Acceptance criteria
1. A `content_update` payload with `section: "notebook_wiki"` replaces only the `## Executive Summary` and `## Notebook Narrative` sections of the target page; content outside those headings (including a manually-added `## KB Notes` section) is byte-for-byte unchanged.
2. A `content_update` payload with `section: "exptube_analysis"` replaces only `## ExpTube Analysis`, independent of whether a notebook content sync has run for that page.
3. First-ever content sync for a page (still holding the empty lifecycle-scaffold body) appends the owned section in the correct fixed position rather than erroring on a missing heading.
4. A payload with `generated_at` older than the last-applied timestamp for that section is rejected with `{applied: false, reason: "stale"}` and does not modify the page.
5. A payload with `content.markdown` exceeding 200,000 characters is rejected with HTTP 413 before any page mutation.
6. Debounced notebook pushes coalesce: 5 rapid `wiki.md` writes within the debounce window produce exactly one outbound `content_update` call.
7. Existing lifecycle-only `create`/`update`/`archive`/`delete`/`restore`/`purge` actions on `/api/sync/experiments` continue to pass their existing tests unmodified — the schema extension is additive.

## Regression risks
- **Tiptap doc parsing brittleness.** If a user manually restructures headings (renames `## KB Notes` to something else, or deletes a section heading entirely), the section-locate step could silently append content in the wrong place or fail to find its anchor. Mitigate: treat a missing expected heading as "append at end, log a warning" rather than throwing, and surface an "unsynced structure" flag in the page's metadata so a human notices.
- **Payload size growth from unbounded LLM narratives.** Mitigate: the 200,000-char cap (AC5); consider truncating with a "content truncated" marker rather than hard-rejecting once real usage data is available.
- **Out-of-order delivery from the a71-01 DLQ retry mechanism.** Mitigate: the `generated_at`-based staleness guard (AC4) is specifically designed to make this safe.
- **New `PageSyncSectionState` model is a live schema migration.** Mitigate: additive table/model only, no changes to existing `Page` columns beyond the metadata this story adds.
- **Two `generateExperimentKbMarkdown()` copies must agree.** If the reconcile-path copy diverges from the route copy, pages that first appear via reconcile break section-merge. Mitigate: update both, or (better follow-up) de-duplicate into one shared helper both call — but de-duplication is a behavior-touching refactor and should be its own small story, not folded silently here.
