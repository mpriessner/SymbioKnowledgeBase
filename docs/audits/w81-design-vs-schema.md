>

But W81 assumes an **a71-13 foundation that has not landed in this repo**, and its two most load-bearing integration points — the `Page.properties` JSONB column and the `a71-13` enrichment engine (`IngestLedgerEntry`, `enrichmentAgent.ts`, `applyPlan.ts`, `ConceptAction`) — **do not exist here**. As written, W81 cannot be migrated or implemented against current `main` without first landing (or absorbing) a71-13.

## Detailed reconciliation

### C1 — CRITICAL — `Page.properties` does not exist
W81-B1 and W81-C1 build their entire "knowledge status" layer on a `Page.properties` JSONB column: C1 is declared "sole owner of `Page.properties.knowledgeStatus`" (w81-c1:22,92), B1 emits a `contested` signal into it (w81-b1:39,91), and both insist all writes go through `jsonb_set` on a subkey and must avoid bumping `Page.updatedAt` (epic-index:83; w81-b1:108,113; w81-c1:79,94,105,117).
**Reality:** the `Page` model (`schema.prisma:183-233`) has no `properties` column. Its JSON-ish fields are `oneLiner`, `summary`, `summaryUpdatedAt` — no generic `properties` bag. The only `properties Json @default("{}")` in the whole schema is on **`DbRow`** (`schema.prisma:347`), an unrelated model. **Impact:** every `jsonb_set(pages.properties, ...)` in B1/C1 targets a non-existent column and fails. The design assumes a71-13 added `Page.properties`; it has not landed here. **Resolution:** W81 must either (a) add `Page.properties JSONB` in its own A-phase migration and stop attributing it to a71-13, or (b) block on a71-13 actually landing. Also reconsider whether page-level knowledge status belongs in a loose JSON bag versus typed columns, given B1/C1's strict "don't touch `updatedAt`" requirement already forces raw-SQL `jsonb_set` gymnastics.

### C2 — CRITICAL — the a71-13 enrichment foundation is absent / mis-shaped
W81-A1/A2 depend on a71-13 as their upstream: `Source` ingestion feeds the "existing" `enrichmentAgent.ts`/`applyPlan.ts` apply loop, dedup is delegated to an `IngestLedgerEntry` table that "short-circuits before the LLM," and claims are emitted via a `ConceptAction` Zod schema and the `/api/agent/pages/enrich` endpoint (w81-a1:12,48; w81-a2:9,21,61,64,77-78,89-95; epic-index:83,85).
**Reality:** none of these schema artifacts exist. There is **no `IngestLedgerEntry` table** (grep of `prisma/` is empty), no `concepts`/`claims`/`sources` tables, and no `src/lib/agent/enrichment/` path. What *does* exist is `src/lib/chemEln/enrichment/` — an **expertise/who-has-experience computation** layer (`expertiseComputation`, `whoToAsk`, `keyLearnings`, `tenantIsolation`) that writes stub one-liners into pages (`src/lib/chemEln/sync/resolver.ts:186-195`), not a source→chunk→claim ingest engine. **Impact:** "W81 builds on a71-13" conflates two different things. The a71-13 that shipped here is chemistry-expertise enrichment; the a71-13 W81 describes (OKF ingest ledger + ConceptAction apply loop) is not present in this repo's schema or code. **Resolution:** confirm where the OKF-flavored a71-13 actually lives (per memory, likely an unmerged branch/other repo), and treat its tables (`IngestLedgerEntry`, any concept store) and endpoints as **net-new prerequisites** W81 must either import or specify itself — not as existing infrastructure.

### H1 — HIGH — migration chain has no a71-13 predecessor
W81 declares a strict, immutable migration order in which a71-13 is the migration that introduces `Page.properties` and `IngestLedgerEntry`, and A1/A2/B1/C1 sequence after it (epic-index:85; w81-a2:94-95). **Reality:** `prisma/migrations/**` ends at `20260617000000_fix_schema_drift` (a reconciliation migration, no new subsystem); there is no a71-13 migration. **Impact:** whoever runs W81's migrations against current `main` has no predecessor that created `Page.properties`/`IngestLedgerEntry`, so any W81 migration referencing them fails; and the "immutable ordering" contract can't be honored because its first link is missing. **Resolution:** fold the missing a71-13 schema deltas into W81's own A-phase, or gate the epic on a71-13's migration landing first.

### H2 — HIGH — `Source`/`SourceKind` naming collides semantically with existing `SourceType`
W81 introduces a `Source` model and a `SourceKind` enum (w81-a1:19-29). **Reality:** the schema already has `enum SourceType {URL, PAGE, MACHINE_PROTOCOL}` (`schema.prisma:75`) used by `DocumentSubscription.sourceType`, plus `sourcePageId`/`sourceUrl` (`:524-526`), `changeSource` on `DocumentVersion` (`:498`), and `SourcePage`/`TargetPage` relation names on `PageLink` (`:270-274`). **Impact:** not a hard DB collision (Prisma allows `Source` model + `SourceType` enum + `SourceKind` enum to coexist), but "source" is already an overloaded term meaning *subscription origin* throughout the codebase. A new top-level `Source` entity plus a near-synonym `SourceKind` enum sitting next to `SourceType` is a maintainability foot-gun — reviewers and future queries will confuse `Source` (W81 immutable document) with `sourcePageId` (subscription origin). **Resolution:** consider a more specific name (e.g. `IngestSource` / `DocumentSource`) and align the enum name.

### M1 — MEDIUM — soft-delete is not schema-wide
W81's scans, supersession, and citation logic assume soft-deleted rows can be uniformly excluded via `deletedAt` "broadly across Page/Claim/Block" (epic-index:86; w81-b1:37,90; w81-c1:27). **Reality:** only `pages.deletedAt`/`deletedBy` (`schema.prisma:201-202`) and `blocks.deletedAt` (`:249`) exist; every other model (Teamspace, Database, DbRow, User, DocumentVersion, FileAttachment…) hard-deletes via Cascade FK. **Impact:** W81's new tables (`Source`, `SourceChunk`, `Claim`, `ClaimEvidence`, …) get no `deletedAt` for free — each must define its own, and any "exclude soft-deleted" join that reaches a non-Page/Block table has no column to filter on. **Resolution:** explicitly add `deletedAt` to each new W81 table that needs it, and don't assume a uniform soft-delete predicate across relations.

### M2 — MEDIUM — polymorphic `originRef` with no integrity, on a nullable key
`Source.originRef` is a polymorphic pointer to `FileAttachment.id | Page.externalId | url` (w81-a1:26). **Reality:** `pages.externalId` is nullable (`schema.prisma:186`, unique-per-tenant but optional), so a `Source` whose `originRef` names a page's `externalId` is unresolvable whenever that page has none; and as a polymorphic string it carries no FK integrity (a deleted FileAttachment/Page leaves a dangling ref). This mirrors the existing no-FK `DocumentSubscription.sourcePageId` (`schema.prisma:526`) — so it's *consistent* with a precedent, but inherits the same dangling-reference weakness. **Resolution:** either store a typed `originKind` + typed id with real FKs where possible, or document that `originRef` is best-effort and handle unresolved refs.

### M3 — MEDIUM — pgvector prerequisite unmet
`SourceChunk.embedding vector(1536)` needs pgvector (w81-a1:46,54,78). **Reality:** the current DB image is plain Postgres 18, not `pgvector/pgvector:pg18`; the extension isn't enabled. The design flags the image swap as a data-volume migration risk — confirmed real: this is an infra prerequisite, not a code detail, and it blocks A1 until the extension is available. **Resolution:** sequence the pgvector enablement (extension + image) as an explicit A0 step with a data-volume backup, or make `embedding` genuinely optional (it's already nullable) and defer vector search.

### L1 — LOW — don't write `Block.search_vector` directly
`blocks.search_vector` (`schema.prisma:246`, `Unsupported("tsvector")?`) is maintained by trigger `blocks_search_vector_update()` from `plain_text` (`migrations/20260221000000_add_search_vector/migration.sql:11-23`). Any W81 worker that writes `search_vector` directly will be overwritten by the trigger. Also its GIN index was dropped in the teamspaces migration (`20260223153420...:8,11`) and not re-declared in `schema.prisma`, so the current FTS index state is ambiguous — **verify against a live DB if W81 relies on FTS ranking.**

### L2 — LOW (validated integration risk) — version pruning vs. cited versions
`Claim.documentVersionId` FK `onDelete:Restrict` (w81-a2:34,146) targets `DocumentVersion`, which is **confirmed real** with `@@unique([pageId,version])` (`schema.prisma:486-509`) — so the FK is valid. The design correctly anticipates that `pruneOldVersions` (retention 100) will hit the Restrict FK once a claim-cited page exceeds 100 versions and roll back the save. This isn't a schema contradiction — it's a real integration hazard the design already prescribes a fix for (skip cited versions in prune). Flagged so implementation doesn't drop that step.

### L3 — LOW (verify in code) — unconfirmed code-file assumptions
W81 asserts specific behaviors of `versioning.ts::createDocumentVersion()` (self-opened `$transaction`, can't be nested), `pruneOldVersions`, `aggregationRefresh.ts` (in-process Map+setTimeout, multi-worker-broken), `scripts/agent-sweep.ts` (budget convention), `duplicatePage.ts` (copies neither properties nor claims), `pageTree.ts` (`updatedAt > summaryUpdatedAt` heuristic), and `withAgentAuth`. These are code-level, not verifiable from schema; the existence of `DocumentVersion` and `SweepSession` (`schema.prisma:637-653`) corroborates the version-history and sweep subsystems are real, but the exact signatures/behaviors need a code-reading pass before implementation relies on them.

## Confirmed CONSISTENT (design matches reality — safe to build on)
- **`DocumentVersion`** model + `@@unique([pageId,version])` (`schema.prisma:486-509`) → `Claim.documentVersionId` FK target is valid; page-level version chain is real.
- **`Block.content`** (Tiptap `Json`) + **`plainText`** (`schema.prisma:240-243`) → A2's text-anchor re-matching against `plainText` has real columns.
- **`FileAttachment.checksum`** (`schema.prisma:562`, nullable) → `Source` dedup/originRef against attachments works, but treat `checksum` as possibly-null.
- **`Page.updatedAt` + `summary_updated_at`** (`schema.prisma:197,214`) → the stale-summary heuristic B1/C1 must avoid tripping is real.
- **`spaceType` enum** `{PRIVATE, TEAM, AGENT}` (`schema.prisma:61`) + **Teamspace** + **PageShare** → W81-D1's Wiki-vs-Brain scoping via `spaceType` has real support.
- **All new W81 model names** (`Source`, `SourceChunk`, `Claim`, `ClaimEvidence`, `EnrichJob`, `ClaimSupersession`, `TriageRun`, `TriageFinding`, `SourceRelevance`) — none exist today, so **no hard table/model collision** (the only concern is the `SourceType`↔`SourceKind` semantic overlap, H2).
- **Tenancy = per-query filter, no RLS** (confirmed `schema.prisma:2`, CLAUDE.md) → W81's insistence on in-transaction tenant-consistency checks for every cross-relation write is **correct and necessary**, not over-engineering. (One quirk: `TeamspaceMember` has no `tenantId` column — `schema.prisma:389-405` — tenant scoping there goes via the `Teamspace` relation; relevant if any W81 join touches teamspace membership.)

## Coverage / limits
- This is a **schema/migration** reconciliation. Code-level assumptions (L3) were spot-checked (confirmed `Page.properties` absence, a71-13 path) but not exhaustively read — recommend a follow-up code pass on `versioning.ts`, `chemEln/enrichment/`, and `aggregationRefresh.ts` before implementation.
- The whereabouts of the "OKF-flavored a71-13" (with `IngestLedgerEntry`/`ConceptAction`) is **unresolved** — it is not in this repo. Locating it (branch or sibling repo) is the single most important prerequisite to make C2/H1 actionable.
- The two W81 docs not read in full this pass (D1/D2, referenced only via the epic index) were not reconciled; their `spaceType`/`summary config` reuse claims check out against existing schema, but a dedicated pass is advisable if they add columns.
